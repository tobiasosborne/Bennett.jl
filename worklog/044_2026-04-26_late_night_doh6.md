**Gotchas / Lessons:**

- **Self-trip on a regression test.** First Pkg.test run failed because the make.jl HEADER COMMENT said "no deploydocs, no GitHub CI per CLAUDE.md §14" and the regression test asserted `!occursin("deploydocs", src)`. The bare-word match fired on the comment. Fix: tightened to `!occursin("deploydocs(", src)` — function-call shape, not narrative mention. *Pattern*: static-inspection tests for "this code shape must NOT appear" should match patterns that distinguish code from comments. Bare-word contains is too coarse; matching the call's `(` is the cheap escape.

- **Two parallel Julia processes is wasteful.** Kicked off `julia --project -e 'using Pkg; Pkg.test()'` and `julia --project=docs docs/make.jl` simultaneously. User flagged it. Both load Bennett independently → 2× RAM, 2× LLVM precompile cost, and they probably contend on the CPU. *Pattern*: serialize Julia jobs unless they're (a) genuinely independent AND (b) the parallel cost beats the sequential wait. Here both criteria failed: docs build was 1-2 min, Pkg.test was 3-4 min — the parallel win was at most 1-2 min and the cache thrash made both slower.

- **Documenter v1's `warnonly` is the right knob for partial coverage.** `warnonly = [:missing_docs, :cross_references, :docs_block]` keeps the build green even if some exported functions aren't in `@docs` blocks (Bennett.jl exports many internal-ish helpers like `pebbled_bennett`, `eager_bennett` etc. that aren't on api.md or reference.md yet). Without `warnonly`, every unrendered export would error the build. `checkdocs = :none` similarly disables the doc-coverage check. These are the conservative defaults for "partial Documenter coverage that we'll grow over time" — opposite of the strict-mode defaults Documenter ships with.

- **Following `@ref` links matters when wiring `@docs` blocks.** First docs build emitted: `Cannot resolve @ref for [`peak_live_wires`](@ref) in docs/src/reference.md`. The warning came from a rendered docstring (one of `gate_count`/`depth`/etc.) that contains a `[`peak_live_wires`](@ref)` markdown link to a peer function. peak_live_wires wasn't in my initial @docs block → no link target. Fix: added it to reference.md. *Pattern*: when adding `@docs` for a function, also add anything its docstring `@ref`s, otherwise Documenter warns.

- **`docs/Manifest.toml` and `docs/build/` are auto-gitignored.** Root `.gitignore` has `Manifest.toml` (matches both root and nested) and `build/` (matches `docs/build/`). So `git add docs/Project.toml docs/make.jl docs/src/...` per-file was the right pattern; `git add docs/` would have ignored the artifacts anyway but per-file is more transparent.

- **Pre-existing tutorial.md → ../../BENCHMARKS.md link generates a Documenter warning.** "invalid local link/image: path pointing to a file outside of build directory." Not introduced by doh6 — the link existed before. Demoted to warning by `warnonly`. Two clean-up paths: (a) copy `BENCHMARKS.md` into `docs/src/benchmarks.md` and rewrite the link, or (b) rewrite the link as an external GitHub URL. Either is a one-liner; filing this as a follow-up suggestion for the next session.

**Rejected alternatives:**

- **Replace api.md's manual prose with `@autodocs` / `@docs`.** api.md has curated organisation (strategy kwarg explanations, supported-input-types tables, prose context that goes beyond docstrings). Replacing it would lose that. The separate `reference.md` page preserves both — curated prose on api.md, auto-rendered + executable doctests on reference.md. Two pages serve two audiences.

- **Add Documenter to the project's test extras + run doctests inside Pkg.test.** Would tightly couple the doctest surface to `Pkg.test()`. Documenter is a heavy dep; the make.jl invocation already provides the regression surface without bloating Pkg.test. Static-inspection test catches scaffold drift; make.jl catches doctest drift. Two-tier coverage with no test-target bloat.

- **Conditional `deploydocs(...)` behind an env var** (e.g. `if get(ENV, "BENNETT_DEPLOY_DOCS", "0") != "0"; deploydocs(...); end`). Tempting flexibility but CLAUDE.md §14 explicitly forbids GitHub CI / remote deploys; the env-var hatch is a footgun. Omitting `deploydocs` entirely is the right call.
