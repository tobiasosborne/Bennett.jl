## Session log — 2026-05-16 — Bennett-zxhg / U_ ConstantStruct global extraction (3+1 — implementer)

**Shipped:** see git log around HEAD; `_extract_const_globals` in `src/extract/module_walk.jl` extended with three dispatch arms (was: single `init isa LLVM.ConstantDataArray || continue` filter). New arms: (a) `ConstantStruct` on `StructType` calls new helper `_flatten_struct_to_bytes` and stores `(UInt64.(bytes), 8)` byte-granular; (b) `ConstantAggregateZero` on `StructType` stores `(zeros(UInt64, abi_size), 8)`; (c) `ConstantAggregateZero` on `ArrayType<IntegerType>` stores `(zeros(UInt64, n), elem_width)` — preserves natural elem_width for the array-zero case, mirroring the original ConstantDataArray path's elem_width inference. New helper `_flatten_struct_to_bytes(init, struct_ty, dl, depth=0) -> Union{Nothing, Vector{UInt8}}` walks `LLVM.elements(struct_ty)` × `LLVM.operands(init)` in source order, dispatching on field type: `IntegerType` (8/16/32/64) → LSB-first byte pack via `LLVMConstIntGetZExtValue`; `ArrayType<IntegerType>` → walk elements via `LLVMGetElementAsConstant`, LSB-first pack each; nested `StructType` → recurse with depth+1; ConstantAggregateZero on sub-field → no-op (bytes already zero); anything else (PointerType, FloatType, VectorType, opaque, undef, IntegerType > 64) → return `nothing` (silent skip in dict, breadcrumb fires downstream). Uses `LLVM.offsetof(dl, struct_ty, i)` + `LLVM.abi_size(dl, struct_ty)` so non-packed structs ABI-padded correctly in the same codepath as packed. Endianness asserted little-endian at helper entry. Depth-limited at 8. G5 message in `src/extract/instructions.jl:483-498` rewritten: (a) clause now reads "ConstantStruct with a non-integer field (e.g. ptr-typed, FloatType, VectorType, or IntegerType wider than 64 bits) — tracked in `Bennett-zxhg-ptrfield`; this is what t5_tr2_hashmap.ll:153's `<{ ptr, [24 x i8] }>` global hits". No signature changes — Q04a contract preserved trivially.

**Gotchas / Lessons:**

1. **LLVM 18 wraps whole-struct `zeroinitializer` as `ConstantAggregateZero`, NOT a `ConstantStruct` of zero fields.** Verified empirically before implementing: `@gz = ... constant <{ [4 x i8], [4 x i8] }> zeroinitializer` parses as `ConstantAggregateZero`. The synthesised design predicted this (arm #3) and the empirical probe confirmed. **Take-home:** the third dispatch arm is load-bearing — without it, every whole-struct zeroinitializer (common in Rust panic-Location ABI) would silently skip. Also added a companion sub-arm for `ConstantAggregateZero` on `ArrayType<IntegerType>` for the same reason — an all-zero `[N x iM]` array would otherwise hit `init isa ConstantDataArray` as false and slip through.

2. **Ptr field with NON-NULL pointee is the real t5 case, NOT a `ConstantStruct` of all zeros.** Initial probe with `{ ptr null, ... }` would have been misleading — `ptr null` operand is a ConstantPointerNull which we could `nothing`-reject easily. The actual t5_tr2:153 global has `ptr @alloc_d077...` — operand kind is `GlobalVariable` (a constant pointer-to-global). Verified with a standalone parse of the relevant 2 globals. The helper's `else return nothing` arm catches it because `field_ty isa LLVM.PointerType` doesn't match any of the IntegerType/ArrayType/StructType arms, propagating the ptrfield breadcrumb correctly. **Take-home:** when synthesising a rejection path, verify it against the actual hostile input, not a simpler proxy.

3. **Float literals in LLVM 18 .ll files must be hex form.** Wrote `float 3.14` in the float-reject fixture; LLVM rejected with "floating point constant invalid for type". Fixed to `float 0x4009000000000000` (0x4009000000000000 is the double-precision representation of approximately π, which LLVM canonicalises for float). Caught immediately on the first fixture-load attempt; not load-bearing for the zxhg logic but worth pinning.

4. **`Bennett-zxhg-ptrfield` is the single breadcrumb for ALL non-integer-field rejects — not separate breadcrumbs for ptr / float / i128 / vector.** This deviates slightly from proposer A's `Bennett-zxhg-nonintegerfield` catch-all naming, but matches the synthesised design (per the orchestrator's spec: "all non-integer / non-array-of-integer fields, returns `nothing`"). The downstream G5 message lists ptr/float/vector/i128 as candidates so the user can identify which subcase they hit. **Take-home:** one breadcrumb per dispatch-leaf is correct here because the eventual `Bennett-land` fix (filed) will address the whole class as one bead; subdividing would create dead breadcrumbs.

## Session log — 2026-05-16 — Bennett-doih / U_ global-pointer src memcpy

**Shipped:** see git log around HEAD; `_handle_memcpy_arm` predicate 5 split into 5a (DST-as-global hard reject — kept) and 5b (SRC-as-global → new `_handle_memcpy_global_src` arm). `globals::Dict{Symbol, Tuple{Vector{UInt64}, Int}}` threaded `_module_to_parsed_ir_on_func` → `_convert_instruction` (new kwarg) → `_handle_intrinsic` (new positional with empty default) → `_handle_memcpy_arm` (new positional with empty default) → `_handle_memcpy_global_src`. New arm runs G1-G9 cascade (dst SSA / dst not global / dst alloca-rooted / dst integer ew / src reaches global + extractable / same-width / multiple-of-ew / in-bounds / fresh dst) and emits K element-granular `IRPtrOffset + IRStore(iconst, dst_ew)` pairs pulling per-element constants from `parsed.globals[gname].data`. Mirrors Bennett-9nwt case C shape but per-element values from the global rather than a single broadcast byte.

**Why:** Bennett-8bys Phase 3 sub-bead under Bennett-hao. Pre-doih `memcpy(local, @anon.<global>, N, false)` failed loud at predicate 5 with a `Bennett-8bys` catch-all breadcrumb (`src/extract/instructions.jl:137-151`). The bead's title ("QROM-style fan-out") was a misnomer for the constant-offset case — there's no index, so we just emit static stores. QROM is genuinely needed for the variable-GEP case (filed as `Bennett-ui4f`).

**Gotchas / Lessons:**

1. **Bead acceptance fixture is unreachable by MVP — but both proposers caught this.** The bead text said "t5_tr2_hashmap.ll line 153 compiles end-to-end", but `build/t5_tr2_hashmap.ll:15` shows the global is `<{ ptr, [24 x i8] }>` — a `ConstantStruct` with a leading `ptr` field, NOT a `ConstantDataArray`. `_extract_const_globals` (`module_walk.jl:269`) filters at `init isa LLVM.ConstantDataArray || continue`, silently skipping it. Even if struct-extraction were added, the leading `ptr` field has no Bennett wire semantics. Both Plan-Opus proposers independently flagged this as a BLOCKER (proposer A: claim 3, lines 16-25; proposer B: claim 3, lines 16-25 + lines 159 BLOCKER section). 2. **Threading `globals` as kwarg through `_convert_instruction` avoids the q04a contract trip.** `test_q04a_convert_instruction_contract.jl:58-61` pins `_convert_instruction`'s signature as the positional Tuple `{Instruction, Dict{_LLVMRef,Symbol}, Ref{Int}, Dict{_LLVMRef, Vector{IROperand}}}`. 3. **Const-GEP src needs dual-path detection.** The dispatch site `if _src_reaches_global(src_v)` initially only checked `LLVMIsAGlobalVariable` and `ConstantExpr<GEP>`. The var-GEP reject fixture (`doih_global_src_var_gep_reject.ll`) uses a runtime GEP `getelementptr i8, ptr @gvg, i32 %xext` — that's an `Instruction`-typed GEP, NOT a ConstantExpr. Without an Instruction-GEP arm in `_src_reaches_global`, the case fell through to predicate 6 and failed with the generic "not alloca-backed" 37mt message instead of the precise `Bennett-doih-vargep` breadcrumb. Caught by the test failing with the wrong breadcrumb on first run (2/77 fails). Added a 3rd arm to `_src_reaches_global` to also recurse through `Instruction`-typed GEPs whose base reaches a global; once the dispatch routes to the doih arm, `_global_root_and_offset` returns `nothing` for the variable index and G5 emits the correct breadcrumb. **Take-home:** "reaches a global" is a structural predicate that must handle both compile-time (ConstantExpr) and runtime (Instruction) GEP wrappers; otherwise the dispatch fails open and the user sees a misleading error.

4. **Reused `_alloca_is_fresh` despite its name suggesting "memset-only".** The helper's first arg is typed `LLVM.Instruction` (generic), and the body is freshness-of-alloca semantics — agnostic to whether the gating call is memset or memcpy. Reusing it (G9) closes the 37mt-inherited hazard (37mt has no fresh-dst gate and would XOR-overlay onto pre-existing dst bytes if the user wrote to dst before the memcpy). doih is the first memcpy variant to apply this gate. The 4-arg name `memset_inst` in the helper's parameter is misleading post-doih; left unchanged because (a) the type is generic, (b) renaming would touch 9nwt's call site, (c) the docstring would need rewriting. Drive-by candidate for a future tidy bead but not load-bearing.
