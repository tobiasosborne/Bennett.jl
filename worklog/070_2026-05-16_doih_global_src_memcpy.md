**Gotchas / Lessons:**

1. **`LLVM.ConstantPointerNull` is not a wrapper in LLVM.jl 9 — dispatch via raw value-kind enum.** Initial cut used `field_val isa LLVM.ConstantPointerNull` per the synthesised spec, which crashed with `UndefVarError: ConstantPointerNull not defined in LLVM`. Replaced with `LLVM.API.LLVMGetValueKind(field_val.ref) == LLVM.API.LLVMConstantPointerNullValueKind`. The same gotcha exists in `_ptr_identity` itself (constexpr.jl:131), which uses the value-kind enum — should have copied that pattern from the start. **Take-home:** when a proposer's spec uses `isa LLVM.SomeWrapper`, verify the wrapper exists in the current LLVM.jl version before implementing.

2. **`@gstruct ... <{ ptr addrspace(1) null, [8 x i8] zeroinit }>` parses as `ConstantAggregateZero`, not `ConstantStruct`.** LLVM coalesces whole-struct all-zero initializers (including `ptr null + zero tail`) to a single `ConstantAggregateZero` regardless of the field types. So an addrspace-only reject fixture that uses `null` ptr + zero tail will bypass the PointerType arm entirely (hits the `ConstantAggregateZero` arm in `_extract_const_globals`, which doesn't recurse into field types). Fix: addrspace reject fixture uses a non-null pointee AND non-zero tail to keep the LLVM parser on the `ConstantStruct` path. **Take-home:** when writing a reject fixture for a specific guard, ensure the input actually reaches that guard's dispatch arm before asserting on the breadcrumb.

3. **Load-escape guard is alloca-level coarse — positive fixtures must use carry-through bodies.** Initial positive test fixtures included a `load i8` of the materialised bytes for oracle purposes (asserting "bytes round-trip through the alloca"). This trips the load-escape guard by design (the only safe consumer pattern is `llvm.memcpy.*`). Restructured: positives are EXTRACTION-only (assert on `parsed.globals` + `parsed.synth_ptr_provenance` directly, no `reversible_compile`), and end-to-end compile is tested via a `carry_through` fixture whose body is `memcpy global→a; memcpy a→b; memcpy b→c; ret i8 %x` with no intermediate load. **Take-home:** when adding a load-escape guard, the natural "load for oracle" test pattern conflicts with the guard — test extraction correctness statically and compile correctness via a load-free fixture. Filed `Bennett-land-precise-escape` (Bennett-4rl0) for the byte-precise refinement that would let oracle loads of the zero-tail region succeed.

4. **q04a contract preserved trivially because all new threading is via kwargs.** `_convert_instruction`'s positional contract `Tuple{Instruction, Dict{_LLVMRef,Symbol}, Ref{Int}, Dict{_LLVMRef, Vector{IROperand}}}` pinned at test_q04a:60-61. Added `synth_ptr_provenance` and `synth_ptr_allocas` as kwargs with empty defaults, same as the existing `globals` kwarg. Same pattern propagated down `_handle_intrinsic` → `_handle_memcpy_arm` → `_handle_memcpy_global_src`. q04a 9/9 green post-edit. **Take-home:** when extending a function with a pinned positional contract, kwargs-with-defaults are zero-impact.

5. **Carry-through propagation in the standard memcpy arm is load-bearing for end-to-end.** The HashMap::new pattern at t5_tr2:145-161 is `memcpy %_3 ← @anon` (tags %_3) → `memcpy %_2 ← %_3` (must tag %_2) → `memcpy %_0 ← %_2` (must tag %_0). Without propagating the tag in the standard memcpy arm (which I initially missed — only added the initial tag in the global-src arm), a downstream load through %_2 / %_0 would silently miscompile. Added a 3-line propagation: if `src_root in synth_ptr_allocas`, `push!(synth_ptr_allocas, dst_root)`. **Take-home:** any side-channel state propagated through a memcpy chain needs explicit forwarding at each link.

6. **`extract_parsed_ir_from_ll(build/t5_tr2_hashmap.ll, ...)` hits an LLVM parse error.** The build/ snapshot was generated with a newer LLVM (`trunc nuw i8 %x to i1` syntax is LLVM 19+; our toolchain is LLVM 18). Added an `:upstream_parse_error` outcome to the T5 acceptance test so an LLVM-version skew between the build/ snapshot and the local toolchain doesn't trip the bead's acceptance — the spec explicitly allows EITHER outcome (compile-end-to-end OR fail-loud), and an upstream parse failure is neither but is an environmental issue orthogonal to this bead. **Take-home:** when an acceptance test depends on an artifact that may drift out of sync with the local toolchain, add an explicit "upstream skew" outcome so the test fails for our reasons, not LLVM-version reasons.

**Rejected alternatives:**

- **A's hash-based address scheme.** Initial proposal hashed `gname` into the low 32 bits of the synthetic address (`BASE | (counter << 32) | (hash(gname) & 0xFFFFFFFF)`). Picked B's pure-counter `BASE | counter` instead per synthesis spec because uniqueness is guaranteed by construction (no birthday collisions across the ~28 t5 globals or the ~4096 plausible-module budget) AND the address is deterministic across re-extractions of the same module (pinned by the new `synth-addr assignment is deterministic` regression test).
