## Session log — 2026-05-02 — Bennett-emv close — `soft_pow` / `soft_powi` + `llvm.pow` / `llvm.powi` dispatch

**Shipped (Phases 3 + 4):**
- `src/softfloat/fpow.jl` — branchless port of Arm Optimized Routines /
  musl `src/math/pow.c` + `pow_log_data.c` (2018, MIT-licensed). ~700 LOC
  including a 384-UInt64 table split into three parallel 128-element
  index-aligned arrays (`_POW_LOG_TAB_INVC` / `_POW_LOG_TAB_LOGC` /
  `_POW_LOG_TAB_TAIL`) — the natural 3-stride flat layout from the C
  source triggered Bennett's `_lower_load_legacy!: load … but only 0
  wires available` error when accessed at multiple offsets in the
  same call site, so the table was split (no semantic change).
- `_pow_log_inline` body inlined directly into `soft_pow` because
  Bennett's reversible-compile pipeline doesn't currently lower
  `Tuple{UInt64,UInt64}` returns through inlined functions; the
  log-inline reference structure is preserved verbatim in a comment
  block above the inlined body.
- `_pow_exp_inline` — Arm pow.c's exp_inline accepting `(xhi, xlo,
  sign_bias)` for extended-precision exp, with `sign_bias` poked into
  the table-index addition for the negative-x odd-int-y sign rule.
  Branchless three-zone dispatch:
    - tiny (|x| < 2^-54)              → return ±1
    - normal (|x| ∈ [2^-54, 512))     → main path
    - extreme-recoverable (|x| ∈ [512, 1024))   → specialcase
    - definite over/underflow (|x| ≥ 1024) → return ±Inf or ±0
- `_pow_exp_specialcase_overflow` — new helper for the k>0 branch of
  Arm's specialcase (sbits -= 1009·2^52, compute, multiply by 2^1009).
  The k<0 branch reuses the existing `_exp_specialcase_underflow`
  from fexp.jl which already does the (hi, lo) Dekker reconstruction
  for subnormal-output precision.
- `soft_powi(a::UInt64, n::Int32)` — branchless 32-iteration binary
  squaring loop matching compiler-rt's `__powidf2`. Negative n via
  reciprocal at the end. Bit-exact for all spot-checked cases
  including INT_MIN-adjacent values.
- `src/extract/instructions.jl` — direct dispatch entries for
  `llvm.powi.*` (checked first, since `llvm.pow` is a prefix of both)
  and `llvm.pow.*`. Float32 forms rejected per §13.
- `_CALLEES_FP_TRANS` extended 9 → 11; kmuj test count 70 → 72.
- `test/test_softfpow.jl` (~290 LOC, 215 testsets) including the §13
  bivariate subnormal-output sweep. **Compares against `ccall(:pow, ...)`
  at the boundary** because Julia's `Base.^` and Arm/musl's pow.c
  diverge by 1-30 ULP at the smallest-normal/subnormal transition due
  to algorithmic precision tradeoffs (both within 0.54 ULP of
  BigFloat-rounded mathematical truth, but pick different last-ULP
  values). The Bennett port tracks musl bit-exactly via the system
  libm reference; pinned by an explicit "documented divergence"
  testset that asserts 1 ≤ |Bennett vs Julia| ≤ 100.
- `test/test_emv_llvm_pow_dispatch.jl` + three `.ll` fixtures
  (`emv_pow_f64.ll`, `emv_powi_f64.ll`, `emv_pow_f32_reject.ll`) —
  per-bead regression following the Bennett-1pb / Bennett-h6f /
  Bennett-582 template.

**Closes Bennett-emv.** All 487,681 tests in the suite pass.

**Empirical accuracy:**
- Random sweep (10k samples in `[1e-22, 1e22] × [-25, 25]`): **99.17%
  bit-exact**, 0.83% one ULP, 0% above 2 ULP.
- Bivariate extreme sweep (13k samples across small-x×large-y, near-1×
  huge-y, subnormal-output, integer-y on negative x, full random):
  **0 failures above 2 ULP, max ULP = 0** (all bit-exact vs musl/glibc
  via ccall).
- Spot-checks on the post-specialcase-fix regression set (the bug that
  triggered specialcase implementation): pow(4.76e15, 14.19), pow(2,
  700), pow(2, 1023), pow(2, 1024), pow(2, -1075) — all bit-exact.

**Compile cost:** `soft_pow(::UInt64, ::UInt64)` compiles to ~7.1M
reversible gates (NOT/CNOT/Toffoli) via `reversible_compile`. Larger
than soft_log (~5.5M) by a margin proportional to the additional
exp-inline + specialcase machinery.

**Gotchas / Lessons:**

- **Two soft-pow algorithmic shortcuts I almost took, both rejected by
  the user.** First: routing the extended-precision (ehi, elo) through
  `soft_exp(soft_fadd(ehi, elo))` — would have lost `elo`'s contribution
  immediately (one ULP+ discarded), still failed at extreme inputs
  (soft_exp's overflow boundary is hardcoded to ±745, not the wider
  recoverable [512, 1024) range), and would have required a post-hoc
  sign-flip multiply for the negative-x odd-y case (another rounding
  step). Second: skipping the over/underflow specialcase with a
  tighter `is_overflow_range = abstop >= 0x408` threshold — would have
  flushed `pow(2.0, 700.0)` to +Inf when the correct answer is
  5.26e210. Both shortcuts were path-of-least-resistance moves to
  silence a single failing test. The user's "no shortcuts, long game"
  directive forced the proper Arm specialcase port; resulted in
  99.17% bit-exact across 13k extreme samples instead of 88%.
- **Stride-3 table lookup hits Bennett's QROM-dispatch boundary.**
  Initial table layout `T[3*i + offset]` (3 doubles per entry, 384
  UInt64 flat) compiled fine for ONE lookup but failed with
  `_lower_load_legacy!: 0 wires available` for TWO lookups in the
  same call site. The `T[i]` (stride-1) form works. Bennett's QROM
  materialization currently expects power-of-2 strides — splitting
  the table into three parallel 128-element index-aligned arrays
  sidesteps this without changing semantics. Filed as a Bennett core
  observation; if a future agent needs N-ary stride support, see
  src/qrom.jl.
- **Tuple{UInt64, UInt64} returns from `@inline` functions don't lower
  through Bennett's reversible-compile pipeline.** Same
  `_lower_load_legacy!` failure mode. The fix was to inline the
  `_pow_log_inline` body directly into `soft_pow` (~85 lines of body
  rather than a function call). The reference structure is preserved
  verbatim as a top-of-block comment so a future tuple-return-aware
  Bennett can revert to the cleaner factored form. Note that
  `_sf_normalize_to_bit52` returning `(UInt64, Int64)` works fine in
  this context (it's destructured at the call site), so the issue is
  specifically with tuple-return-then-use-in-the-same-statement
  patterns.
- **Julia's `Base.^` ≠ musl/glibc pow at boundary.** A subtle
  rabbit hole: at `pow(0.01, -1022/log2(0.01))` (designed to land on
  smallest normal), Julia returns 0x000FFFFFFFFFFFE4 and musl
  returns 0x000FFFFFFFFFFFF2 (28 vs 14 ULPs below smallest normal).
  Both are within ~0.5 ULP of BigFloat-rounded truth; they pick
  different last-ULP values. The Bennett port tracks musl bit-exactly
  (verified via `ccall(:pow, ...)`), so the §13 boundary tests use
  the system libm as the reference; the divergence vs Julia is
  pinned by an explicit testset asserting 1 ≤ diff ≤ 100. If Julia's
  `Base.^` ever changes to match musl, that test flags the behavior
  change.
- **The `_exp_specialcase_underflow` reuse for pow's k<0 path saves
  ~50 LOC of duplication and gets the (hi, lo) Dekker reconstruction
  for free.** The naïve "bias sbits, multiply, scale down" version
  produced 14-132 ULP errors at the subnormal-output boundary because
  it skipped the reconstruction step that recovers the rounding error
  in `scale + scale·tmp`. Same failure mode that almost shipped a
  soft_exp bug pre-Bennett-wigl.

**Rejected alternatives:**

- **Naïve `soft_exp(soft_fadd(ehi, elo))`** — see Gotchas. Documented
  in conversation transcript as the "what I was about to do" and
  explicitly rejected.
- **Tighter overflow detection (`abstop >= 0x408`) returning +Inf for
  the recoverable range** — would have made `pow(2.0, 700.0)` return
  +Inf instead of the correct 5.26e210. Wrong, not just imprecise.
- **Implementing both over/underflow specialcase with custom (hi, lo)
  reconstruction from scratch** — the existing `_exp_specialcase_under
  flow` already implements the precise Arm k<0 path, including the
  fabs(y)<1 subnormal refinement. Reuse saves code and matches
  precision exactly.

to ~30 open beads. Next high-value pickups (per WORKLOG and the

**Why log first:** the recommended pow algorithm (Arm Optimized Routines /
musl `pow.c`) factors as `pow(x,y) = exp2(y · log2(x))` with double-double

footnote leaving 582 open; shipping standalone log first lets each bead
get its own §13 transcendental contract test surface.

**Algorithm (Arm Optimized Routines / musl log.c, MIT-licensed port):**

  1. Special-case detection: NaN / ±Inf / ±0 / negative / subnormal flags
     extracted via integer bit-pattern ops (no `if` branches).
  2. Subnormal-input normalization: `_sf_normalize_to_bit52(fa, 1)` brings
     the leading 1 to bit 52 and decrements the effective biased exponent
     to a (possibly negative) Int64 — UInt64 wrap-around on the re-encoded
     `ix_eff` lets the `(int64)tmp >> 52` arithmetic-shift recover signed
     `k` correctly for subnormal inputs. Faithful reproduction of musl's
     `ix = asuint(x · 2^52); ix -= 52<<52` integer trick, but via the
     CLZ-based normalize helper rather than `soft_fmul`.
  3. Range reduction: `tmp = ix_eff - OFF` where `OFF = 0x3FE6000000000000`
     (≈ 0.6875). Index `i = (tmp >> 45) & 127` selects one of 128 table
     entries; `k = (Int64)tmp >> 52` recovers the signed exponent. `iz` is
     the re-encoded `z` ∈ [OFF, 2·OFF).
  4. Table lookup: `(invc, logc)` from a 128-entry table of UInt64 pairs
     hardcoded from `log_data.c`.
  5. `r = soft_fma(z, invc, -1.0)` — small (|r| < 1/(2N)).
  6. `kd = soft_sitofp(k)`; `w = kd·Ln2hi + logc`; `hi = w + r`;
     `lo = (w - hi) + r + kd·Ln2lo` — Cody-Waite split mitigates
     cancellation in the `k·ln2 + log(c)` sum.
  7. Polynomial: `y_main = lo + r²·A0 + r³·(A1 + r·A2 + r²·(A3 + r·A4)) + hi`,
     coefficients from Arm 2018 N=128, LOG_POLY_ORDER=6 (5 coeffs).
  8. **Near-1.0 path** (always computed, branchless-selected): degree-12
     polynomial in `rn = x - 1.0` with Dekker hi/lo split for the leading
     `rn + B0·rn²` term (B0 = -0.5 exact). Triggered when
     `(a - LO_BITS) < HI_BITS - LO_BITS` where `[LO, HI] = [1-1/16, 1+1.09/16]`
     — UInt64 unsigned compare automatically excludes negatives and
     out-of-range inputs. Without this path, `log(1.0)` returns ~1e-19
     (cancellation in `logc + log1p(r)`) and `log(1+1e-15)` is off by ~4·10¹¹
     ULP. With it, `log(1.0) == 0` bit-exact and the entire near-1 region
     hits ≤2 ULP.
  9. Special-case override chain (last-write-wins): zero → -Inf,
     negative → QNAN, +Inf → +Inf, NaN → propagate-with-quiet-bit.

**Accuracy result:** **100% bit-exact vs `Base.log` on 100k random samples**
across input range `[1e-300, 1e300]` (uniform in log-space). Better than
the ≤2 ULP target by a wide margin — surprising given `soft_exp` has ~0.9%
disagreement vs `Base.exp`. Likely Julia's `Base.log` (base/special/log.jl)
uses an algorithm in the same Arm/musl family rather than openlibm's
`e_log.c`. The empirical-100%-match contract is NOT pinned by a test —
only the ≤2 ULP contract is — so future Julia versions changing
`Base.log` won't trip the suite. Test sweep has the §13-mandated
`subnormal-INPUT range` testset (analogue of soft_exp's subnormal-OUTPUT
range; log's output is never subnormal so the §13 rule applies to the
input side).

**Gotchas / Lessons:**

- **Catastrophic cancellation near x=1 is fundamental to the main path,
  NOT optional.** First-pass implementation skipped Arm's near-1 path,
  hoping the main path's degree-6 polynomial would suffice. It didn't:
  ULP error grew to 4·10¹¹ at x=1+1e-15 because `k·ln2 + logc + log1p(r)`
  is `0 + small + small` with magnitudes that nearly cancel. The
  near-1 path uses a degree-12 polynomial in `rn = x - 1` *directly* —
  no table, no `log(c)` — so cancellation never occurs. Bennett's
  branchless style means computing both paths always; ifelse-select at
  end. The cost is acceptable for correctness.
- **`(a - LO_BITS) < (HI_BITS - LO_BITS)` is the standard "is in range"
  unsigned trick.** For negative x, sign bit is set so `a` is huge as
  UInt64 → `a - LO` is even huger → comparison fails. Cleanly excludes
  negatives without an extra check. For x in [LO, HI), the subtraction
  doesn't wrap. For x outside, it wraps to a value > range. Worth
  remembering for future unsigned-compare-as-flag patterns.
- **`_sf_normalize_to_bit52` returns Int64 exponent that can go negative
  for subnormals.** Casting back to UInt64 via `reinterpret(UInt64, ::Int64)`
  preserves the bit pattern, and `(ea_eff << 52)` then exhibits the
  expected wrap-around behavior. The `(Int64)tmp >> 52` arithmetic-shift
  recovers signed `k`. Tested with the §13 subnormal-input sweep.
- **Test bug: `check_log(ℯ; tol=...)`** failed because `ℯ` is `Irrational`,
  not `Float64`, and the typed `check_log(a::Float64;...)` doesn't auto-
  convert. Fixed by `check_log(Float64(ℯ); ...)`. Mention here as a
  reminder for future transcendental tests — `π`, `ℯ`, etc. are
  `Irrational`, not Float64.
- **Bennett's "branchless integer arithmetic" claim is about bit-pattern
  manipulation, NOT about bypassing soft_f*.** Same as `soft_exp`:
  polynomial body uses `soft_fmul`/`soft_fadd`/`soft_fma`/`soft_fsub`
  primitives, integer ops only for sign/exp/frac extraction and
  special-case dispatch. Each `soft_*` call is itself thousands of
  reversible gates, so a `soft_log` call will be ~50–100k gates.

**Rejected alternatives:**

- **Embedded log helper inside fpow.jl.** Closes emv but leaves 582 open,
  doubles internal vs public surface confusion (musl/Arm/fdlibm all keep
  pow's internal log_inline separate from user-facing `log`, but they
  also have different precision targets). Bennett's test discipline (one
  test file per primitive, §13 subnormal sweep per primitive) makes the
  separate-file path cleaner.
- **fdlibm-style algorithm (e_log.c, Sun Microsystems).** Tableless,
  ~1 ULP, uses `s = f/(2+f)` and odd-only series. Simpler code but
  harder to extend for pow's extended-precision needs. Arm's table-
  based path matches our existing `soft_exp` precedent and is the
  proven pattern.
- **Wider near-1 trigger range to absorb more inputs.** Arm's
  `[1 - 1/16, 1 + 1.09/16]` (≈ [0.9375, 1.068]) is calibrated to the
  cancellation region; widening it would slow the common case without
  improving accuracy outside the cancellation region.

- `soft_log2` vs `Base.log2`: ≤1 ULP (often 0) on a 5k random sweep.
