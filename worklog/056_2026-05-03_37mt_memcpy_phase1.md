**Synthesis decision (3+1 protocol, Proposers A and B):**

Both proposers agreed on the byte-granular `IRPtrOffset+IRStore`
expansion shape (same architectural reason as 37mt — `aggregate.jl:227`
and `memory.jl:245-246` gates). The interesting divergence was
freshness handling for the c≠0 case:

  - **Proposer A: option γ** — ~70-line `_alloca_is_fresh` intra-block
    sweep that walks from the alloca to the memset checking for
    aliasing writes. Conservatively rejects on opaque pointer ops,
    cross-block, etc. Fail-loud where current behaviour silently
    miscompiles.
  - **Proposer B: option β** — assume fresh by Julia/Rust frontend
    convention; document the c≠0 non-fresh hazard. Zero implementation
    cost.

Synthesised: **option γ for c≠0 path** (silent miscompile is real and
§1-relevant), **option β-style early short-circuit for c=0** (predicate
8 returns `IRInst[]` before alloca/freshness checks, preserving the
pre-9nwt benign-allowlist broad tolerance for unaudited Julia frontend
patterns). This is the conservative-but-correct mix: c=0 keeps doing
what it did, c≠0 properly fails loud on non-fresh dst.

Also picked up Proposer B's `llvm.memset.inline.*` catch — the predicate-1
addrspace check accepts both `llvm.memset.p0.*` and
`llvm.memset.inline.p0.*`.

**Predicate cascade (13 predicates, see docstring):** addrspace 0,
op-count, vol const 0, c const, N const, N>=0, **N==0 → no-op**,
**c==0 → no-op (case A early exit)**, dst SSA registered, dst not
global, dst alloca-rooted, alloca elem_w==8, dst alloca fresh
(intra-block sweep). The two no-op short-circuits (predicates 7 and 8)
are deliberately placed before the dst checks — N=0 and c=0 are LangRef
no-ops or pre-9nwt-benign no-ops respectively.

**Gotchas / Lessons:**

1. **Test fixture for "non-alloca dst reject" needs non-void return.**
   First version of `9nwt_memset_param_dst_reject.ll` had `define void
   @memset_param_dst(ptr %p)` calling memset and returning. The IR
   walker errors out on the void return type via `_type_width` (the
   Bennett-dq8l error fires on void return inference) BEFORE reaching
   `_handle_memset_arm`. Fix: have the function `load i8` from the
   pointer post-memset and return the byte. Now the memset call is
   processed in order and the predicate cascade fires correctly.

2. **c=0 short-circuit before alloca checks is intentional.** The
   simplest cascade would run all 13 predicates uniformly. But that
   would tighten c=0 behaviour from "silent drop on any dst" (pre-9nwt
   benign-list) to "fail-loud on non-alloca dst". Without auditing
   every Julia frontend code path that emits memset(0) for non-alloca
   targets, that risks regression. The synthesised cascade ships c=0
   as broadly-tolerant as before; only c≠0 gets the alloca + freshness
   gates.

3. **Same-block freshness only.** `_alloca_is_fresh` returns false for
   cross-block alloca/memset pairs. Julia and Rust frontends emit
   memset in the alloca's block (entry-block GC zeroing), so this is
   fine in practice. Hand-written `.ll` with cross-block memset gets
   fail-loud → 8bys-uncompute, which is correct safety.

4. **Opaque-pointer stores conservatively reject freshness.** Any
   `Store` whose pointer doesn't trace to an alloca via
   `_alloca_root_ref` is treated as a possible write to dst (false
   positive). Reduces silent miscompilation risk at the cost of
   over-conservative fail-loud rate. Acceptable trade.

**Why deviation from the bead's verbatim "(N/8) IRLoad+IRStore pairs at
64-bit granularity, memory.jl needs no change" wording:** independent
investigation by both 3+1 proposers (and confirmed by reading
`src/lowering/aggregate.jl:227` and `src/lowering/memory.jl:245-246`)
revealed that the bead's spec is internally inconsistent. Two gates in
the existing memory pipeline force the design:

  1. `lower_ptr_offset!` (`aggregate.jl:227`) only propagates
     ptr_provenance when `ew == 8` — non-MVP shapes are skipped (a
     `continue`). For `alloca i64` (ew=64), an `IRPtrOffset(off=8)`
     produces a dest with NO provenance, and a subsequent IRStore
     through that dest fails the provenance assertion in
     `lower_store!`.
  2. `_lower_store_via_shadow!` (`memory.jl:245-246`) requires
     `inst.width == elem_w`. A 64-bit IRStore into an `alloca i8`
     slot fails with `DimensionMismatch`.

  → The only chunk shape that lands cleanly through the existing
  memory.jl is **byte-granular (width=8) chunks on `alloca i8`
  (elem_w=8)**. So Phase 1 emits N width=8 chunks instead of (N/8)
  width=64 chunks, and rejects allocas with elem_w ≠ 8 with a precise
  `Bennett-8bys` reference. The bead's "N is a multiple of 8 bytes"
  precondition becomes moot under byte-granular chunks; any positive N
  works.

  This deviation is an honest scope tightening, not scope creep — the
  set of N>8 memcpy patterns where 64-bit-chunk lowering would have
  Just Worked under memory.jl as-is is **empty** (verified
  empirically). 64-bit chunked lowering belongs to Bennett-8bys
  (alongside the memory.jl extensions it requires).

**Empirical coverage:**
- N=4 / N=8 / N=24 byte memcpy compile end-to-end, `verify_reversibility`
  passes, identity oracle bit-exact for `Int8(-8):Int8(8)` (16 inputs ×
  3 sizes = 48 round-trip samples, all clean).
- Reject paths: volatile, variable-N, same-alloca self-copy, alloca-i64
  elem_w mismatch, memmove — each fires its own predicate with the
  right `Bennett-8bys` (catch-all) or `Bennett-haod` (deferred globals
  sub-bead) reference.

**Gate budget (no baseline pinned):** N=24 byte memcpy emits 24
IRPtrOffset×2 + 24 IRLoad + 24 IRStore = 96 IR instructions at width=8.
After lowering through the shadow path (3W CNOTs per width-8 store/load
in the entry-block fast path = 24 CNOTs each), the per-memcpy gate cost
is roughly 24×24×2 ≈ 1150 forward CNOTs, doubled by Bennett's reverse
pass. For the t5 corpus's typical N ∈ {24, 32, 48} this is a few
thousand gates per call — large but tractable. A wider chunk strategy
would cut this 8×; that optimisation lands with Bennett-8bys.

**Gotchas / Lessons:**

1. **The bead spec was internally inconsistent.** "(N/8) at 64-bit
   granularity" + "memory.jl unchanged" cannot both hold given current
   `aggregate.jl:227` (`ew == 8 || continue`) and `memory.jl:245-246`
   (`inst.width == elem_w`). Independent verification by both proposers
   converged on the same diagnosis. Worth remembering: when a bead spec
   sounds frictionless, double-check the lowering invariants it
   implicitly relies on.

2. **`[N x i8]` ArrayType allocas are silently dropped** by
   `instructions.jl:1057` (`elem_ty isa LLVM.IntegerType || return
   nothing`). Phase 1 doesn't fix this — the t5 corpus's
   `alloca [N x i8]` shape would still be silently dropped before the
   memcpy lowering can use it. If raw .ll/.bc ingest needs `[N x i8]`
   alloca extraction, file a follow-up bead. Phase 1 covers
   `alloca i8, i32 N` shape, which is what hand-crafted .ll fixtures
   and some clang outputs use.

3. **Recursion bound on alloca-root walk.** `_alloca_root_ref` recurses
   through const GEPs back to the alloca; bounded at depth 8 to defend
   against pathological IR. LLVM doesn't usually nest GEPs > 2 in
   practice, so the bound is a safety net rather than a tight
   constraint.

4. **`LLVM.opcode(val) == LLVM.API.LLVMGetElementPtr` for GEP detection.**
   Used in `_alloca_root_ref` rather than `val isa LLVM.GetElementPtrInst`
   because LLVM.jl wraps GEP instructions as generic `LLVM.Instruction`
   with no specialised Julia type. The opcode check is the canonical
   pattern in the rest of `instructions.jl`.

5. **Same-alloca check uses ALLOCA root equality, not SSA ref equality.**
   `memcpy(%a, %a, N)` and `memcpy(%a, gep(%a, 0), N)` both alias and
   should both reject. Comparing alloca roots after the GEP-traversal
   walk catches both shapes; comparing operand SSA refs would miss the
   second.

**Rejected alternatives:**

- **64-bit chunks restricted to `alloca i64` allocas only** (Path β
  from Proposer B). Cleaner gate count but doesn't cover the t5 corpus
  shape (`alloca [N x i8]` / `alloca i8`). Picking this would have
  meant Phase 1 ships purely as a contract-honouring no-op for raw
  `.ll`/`.bc` ingest — no actual coverage gained. Rejected.

- **Adaptive chunk width matching the alloca's elem_w** (an extension
  of Path β). Elegant but doubles the implementation surface (separate
  expansion paths for ew=8 and ew=64) and still needs the
  `aggregate.jl:227` gate lifted for ew=64. Lifting that gate is
  Bennett-8bys territory by design — the gate exists for good reason
  (the const-folded provenance update assumes byte-granular indices).

- **Extending `[N x i8]` ArrayType alloca extraction in 37mt** (Path α
  from Proposer A's "sidecar"). Tempting because t5_tr2_hashmap.ll
  uses `[N x i8]` and our Phase 1 doesn't help it without that
  extension. Rejected per CLAUDE.md §11 (PRD-driven; don't expand
  scope) — file a follow-up bead if real-world `.ll` ingest needs it.
  The bead description was deliberately scoped to memcpy lowering, not
  alloca-extraction extensions.

- **Reading the LLVM call-site `align N` attribute on dst/src.** The
