**Algorithm decision (3+1 synthesis).** Two parallel `Plan` proposers
explored the design space. Both converged on a regime-split rather
than a verbatim musl `s_tanh.c` port (which needs `soft_expm1` —
Bennett.jl doesn't have one, and adding it for one bead is scope
creep). Synthesis adopted Proposer B's strategic choice (port Julia
stdlib `Base.tanh(::Float64)`) for these reasons:

1. **Subnormal-input bit-exactness is implicit, not via override.** For
   any subnormal `x`, `x²` underflows to `+0`, `tanh_kernel(0) = 1.0`
   (the constant term of the polynomial), and `soft_fmul(x, 1.0) ≡ x`
   bit-exactly. So `soft_tanh(2^-1075) ≡ 2^-1075` follows by
   construction, no explicit override needed. The §13 contract is
   satisfied through the polynomial branch's algebra.
2. **Vetted polynomial coefficients.** Julia stdlib's `tanh_kernel` is
   a degree-10 minimax in `z = x²` fitting `tanh(x)/x` on `|x| ≤ 1` to
   ≤1 ULP, copy-pasted verbatim from `julia 1.12 base/special/
   hyperbolic.jl:132-138`. Reusing them sidestep an ULP-validation
   round.
3. **`soft_exp_fast` insight (Proposer B).** The exp argument is
   `2|x|` — always non-negative. `soft_exp_fast`'s FTZ branch only
   fires for `x ∈ [-745.13, -708.40]` (negative). So `soft_exp_fast`
   and `soft_exp` are bit-identical here, and `soft_exp_fast` saves
   ~1.4M gates. **Costs zero accuracy.**

Adopted from Proposer A: defensive `llvm.tanh.` arm placement AFTER
`llvm.tan.` arm (defence-in-depth against future prefix relaxation,
even though the trailing `.` from Bennett-7goc already protects), the
strict NaN classification via exponent+fraction split (not high-word
only), and the regression test for tan-vs-tanh dispatch separation.

**Gotchas / Lessons:**

1. **§13 subnormal-input contract via polynomial-branch algebra (no
   override).** Proposer A wanted an explicit `is_tiny → return a`
   override; Proposer B argued it's redundant because the polynomial
   branch handles it implicitly. B was right. Subnormal `x` →
   `soft_fmul(x, x)` underflows to `+0` → `tanh_kernel(0) = P0 = 1.0`
   → `soft_fmul(x, 1.0) ≡ x` per IEEE 754. **The implicit mechanism
   is more robust** because future polynomial-coefficient changes
   can't break the subnormal preservation as long as `P0 = 1.0`
   holds (and it must, by definition of `tanh(x) ≈ x` near origin).
2. **`soft_exp_fast` is bit-identical to `soft_exp` for non-negative
   input.** The FTZ branch only fires for `x ∈ [-745.13, -708.40]`
   (negative input range producing subnormal output). Our
   `soft_exp_fast(2|x|)` always gets a non-negative argument — the
   FTZ branch is unreachable. Saves ~1.4M gates per `soft_tanh` call.
   General lesson: when adding new transcendentals built on `exp`,
   check whether the exp argument has a known sign and prefer
   `soft_exp_fast` over `soft_exp` when the FTZ branch is dead code.
3. **Test fixture entry-function-name pitfall.** The first attempt at
   the regression-guard test used `entry_function="tan_intr"` for
   `s1zl_tan_f64.ll`, but the actual entry function in that fixture
   is `@tan_f64`. Failed loud with an LLVM module-walk exception. Fix
   was a one-character grep at `test_s1zl_llvm_tan_dispatch.jl` to
   confirm the right name. **General rule when reusing an existing
   fixture: grep the existing test file that ships with it for the
   exact `entry_function=` literal — don't guess from the filename.**
4. **NaN classification via exponent + fraction (not high-word).**
   Proposer B's first draft used `is_saturate = abs_a >= _TANH_22_BITS`
   which mis-fires on NaN (NaN bit patterns have `abs > 22.0` in
   unsigned ordering). The fix is the final `is_nan` override at the
   bottom of the cascade — last-write-wins corrects the saturation
   mis-fire. Used Proposer A's `(ea == 0x7FF) & (fa != 0)` check for
   `is_nan`, not the high-word approximation.
5. **Polynomial coefficient ordering for Horner.** Julia stdlib uses
   `evalpoly(z, (1.0, c1, c2, ..., c10))` which is `1.0 + c1·z +
   c2·z² + ...`. Bennett-branchless Horner from inside-out: start
   with `c10`, then `c9 + z·tmp`, ..., end with `c0 + z·tmp = P0 +
   z·(...) = tanh_kernel(z)`. Final assembly: `result = soft_fmul(a,
   tanh_kernel(z))` carries the sign of `x` implicitly. Cleaner than
   computing on `|x|` and OR-ing the sign at the end.

**Rejected alternatives:**

* **Verbatim musl `s_tanh.c` port** — needs `soft_expm1` for the
  small-arg branch. Adding `soft_expm1` would be ~600 LOC (own
  Tang-style table, polynomial, subnormal specialcase, accuracy
  contract, parity tests) — multi-bead scope creep. Substituting
  `soft_exp(2x) - 1.0` would suffer catastrophic cancellation at
  small `|x|`, violating §13.
* **Composed-identity `(e^{2x}-1)/(e^{2x}+1)` everywhere.** Same
  cancellation problem at small `|x|`. Subnormal-input regime
  collapses to ±0 output instead of `±x`, breaking §13.
* **Padé / minimax everywhere with symmetry (no soft_exp).** Tempting
  (cheaper gates) but tanh's asymptotic saturation at `|x| → ∞`
  doesn't admit a clean rational approximation on an unbounded
  interval; needs a regime split anyway. And the hand-fit minimax
  for the medium-magnitude regime would lack a vetted ULP reference.

**Next agent starts here:** **`sinh` is the next pickup** in C1 (Tier
C1.7). Reuses `soft_exp` similarly — `sinh(x) = (e^x - e^{-x})/2`.
Unlike tanh, sinh does NOT saturate; for `|x|` large it overflows to
`±Inf`. Pattern: file the bead, follow the playbook (Julia stdlib
`Base.sinh` regime split — `tiny | small (poly) | medium (exp pair) |
huge (single exp + scale)`). Note: `Base.sinh` uses TWO exp calls in
the medium regime (one for `e^x`, one for `e^{-x}`); consider whether
the same `e^x + 1/e^x` trick applies for cheaper Bennett gate cost,
or whether the symmetry `sinh(x) = sign(x)·sinh(|x|)` allows folding
to one exp via `(e^{2|x|}-1)/(2·e^{|x|})` form.
