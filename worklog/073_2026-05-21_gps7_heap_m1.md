## Session log — 2026-05-21 — Bennett-kuza / gf3n M2 — Julia heap-memory support, Milestone 2

**Mode:** orchestrated 3+1 (CLAUDE.md §2) — sonnet IR-reconnaissance → 2
independent opus `Plan` architects → opus implementer → orchestrator review
(+1). Serial, one subagent at a time.

**Shipped (this commit):** M2 closed. `Array{T}(undef,const_N)` heap integer
arrays whose element data is LIVE into the return now compile under
`mem=:heap`. M2 extends `src/extract/heap.jl` (~+1100 LOC): `_detect_gc_preamble!`
forks — M1 (fully-dead skeleton → drop) vs M2 (partially-dead → re-root). The
M2 path recognises the heap array (`_recognise_heap_array`: 1 alloc = Memory
only / 2 allocs = Memory backing + Array wrapper), **partitions** M1's
monolithic `SKEL` into {dead GC machinery → drop} ∪ {live element traffic →
re-root} (`_partition_skeleton`), runs a 7-obligation bail-loud soundness proof
(`_prove_partition_sound`: M2-O1 store-capture, O2 load-capture, O3
machinery-dead, O4 taint-containment, O5 no-escape, O6 capacity, O7
uniform-width), collapses the Julia bounds-check diamond
(`_collapse_bounds_diamond`), and re-roots the element stores/loads onto a
synthetic `IRAlloca(_, 8, ConstOperand(N))` (`_build_rerooted_slice`) — which
the existing `:shadow`/`:mux`/`:shadow_checkpoint` lowering compiles unchanged.
NO new IR node, NO lowering change, NO `register_callee!` change. Only other
src edit: 1 line threading a `counter::Ref{Int}` through `module_walk.jl`. New
`test/test_kuza_heap_m2.jl` 9734/9734 + `test/fixtures/heap_m2_cond_pair.ll`.

**Why:** M2 is the "toy → real" step of the dynamic-memory arc — it adds
genuine heap-data *re-rooting* (vs M1's drop-only), so a real Julia
`Array{T}(undef,N)` compiles to a reversible circuit.

**Architecture decision (3+1 synthesis).** Both architects agreed on the
SKEL-partition. They diverged on the Julia bounds-check diamond: Architect A —
M2 stays straight-line-only, *reject* the diamond, test `:shadow_checkpoint`
via a hand-written straight-line g256 `.ll` fixture. Architect B — M2 *must*
collapse the diamond. Orchestrator chose **B**, decisively: B proved
empirically that a runtime-indexed access into a non-SROA'd heap array
**necessarily carries a bounds-check diamond** — the diamond is exactly what
blocks SROA; remove it and the array SROAs to a stack alloca (no heap
skeleton). So a straight-line-only M2 compiles *nothing real* beyond cond_pair,
and A's hand-written straight-line g256 tests a shape the Julia compiler never
emits. B's diamond collapse is narrow + sound: the throw arm is
`unreachable`-terminated (no successors ⇒ no phis to rewrite — A overestimated
it as "M3-grade"), recognised structurally (`j_throw_boundserror_*` callee +
`unreachable`), and dropping throw paths is already established Bennett
behaviour (`ijl_bounds_error`/`j_throw_*` are in the project-wide benign-drop
allowlist).

**Gotchas / lessons:**
- **SROA threshold.** `Array{Int8}(undef,N)` for N∈{8,16,64} is fully SROA'd by
  LLVM to a stack `alloca [k x i64]` — NO heap skeleton survives. Only **N≥256**
  keeps a live `@ijl_gc_small_alloc`. The masked-index idea (`(i&127)+1` to
  elide the bounds check) is a dead end — eliding the check lets SROA eat the
  whole array. A runtime-indexed heap-array read is inseparable from its
  bounds-check diamond.
- **The g256 test must use `mod(i,3)`, not `mod(i,100)`.** `mod(i,100)+1` reads
  elements 4..99 which the function never writes — so `g256` *itself* returns
  `undef` heap garbage there, making any oracle non-deterministic. `mod(i,3)+1
  ∈ {1,2,3}` hits only written slots. N stays 256 (index width does not affect
  SROA). General lesson: an oracle test of a heap-array function must only
  sweep indices the function actually writes.
- **M1-vs-M2 routing is on `growend!`, not on element-traffic emptiness.** M1's
  dynamic `Int8[]`+`push!` IR has NO constant Memory-length store (the length
  is a runtime `growend!` result) — so `_recognise_heap_array` would
  mis-reject it. Route: a `@j_#_growend!` callee present ⇒ M1-shaped (dynamic
  vector) ⇒ M1 path; absent ⇒ M2 recognition. Keeps M1's f1 byte-identical.
- **The M2-aware ret-ancestor walk** (`_heap_m2_return_ancestors`) models the
  *re-rooted* return: it stops at `data_ptr` and recurses an element GEP only
  into its index operand (not its base) — because after re-rooting the load
  addresses the synthetic alloca, not the original Memory→alloc chain. A naive
  walk would mark the whole GC chain a ret-ancestor and reject everything.
- `cond_pair` under `--check-bounds=yes` is the simplest M2 shape (1 alloc,
  N=2, straight-line) and is M2's `.ll`-fixture test. Under default
  check-bounds it SROAs (no skeleton) — so the M1 test's cond_pair assertion
  is unaffected and M1 stays 519/519.

**Review fixes (orchestrator +1).** Two issues caught in review, both fixed:
(1) `_build_rerooted_slice` silently handled runtime-index element STORES — but
the consensus design §4 M2 scopes M2 to "constant-offset stores + runtime-
indexed loads"; a runtime-index store was untested AND unproven (the M2-aware
walk vets runtime *load* indices but never reaches store indices). Now
`_partition_skeleton` rejects a runtime-index element store loud; follow-up
filed as **Bennett-5006** (also unblocks the Bennett-890r TJ4 redesign). (2)
`_build_rerooted_slice` would `push!(survivors, nothing)` if `_convert_instruction`
returned `nothing` for a benign non-skeleton intrinsic — now skips it.

**Rejected alternatives:** Architect A's straight-line-only M2 (see Architecture
decision). Runtime-index element stores (deferred to Bennett-5006).

**Regression:** test_kuza_heap_m2 9734/9734; test_gps7_heap_m1 519/519
(byte-identical); test_gate_count_regression 39/39 (baselines unchanged); plus
the implementer's earlier green sweep of universal_dispatch 293/293, increment
257/257, memory_corpus 582/582, mutable_array 577/577, t5_corpus 260/260,
ir_extract, lower. Default `mem=:auto` path provably untouched (all M2 logic
gated on `mem===:heap`). Full `Pkg.test()` was running in background as a
post-M1 foundation check; not gated on for this commit (M2's surface is
regression-covered above and the default path is untouched).

**Next agent starts here:** `Bennett-5ikt` (gf3n M3) — `push!`-built `Vector`
with a statically-inferable element count: handle `j_#_growend!` + the
size-counter + the capacity diamond as dead skeleton, infer capacity from the
constant-offset store set, re-root the surviving element stores/loads. Makes
`f_tj1` (push!×3 + reduce) genuinely green. 3+1 required. heap.jl's M2

## Session log — 2026-05-21 — Bennett-gps7 / gf3n M1 — Julia heap-memory support, Milestone 1

**Mode:** orchestrated 3+1 (CLAUDE.md §2) — sonnet IR-reconnaissance → 2
independent opus `Plan` architects → opus implementer → orchestrator review
(+1). Serial, one subagent at a time.

**Shipped (this commit):** `Bennett-gf3n` epic split into 4 child beads M1-M4
(`gps7`/`kuza`/`5ikt`/`bd5f`); `Bennett-cc0.5` retired as superseded (its
thread_ptr-GEP premise was disproven by the 2026-05-21 spike — the wall is an
inline-asm TLS read). M1 closed: new `src/extract/heap.jl` (~772 LOC) — a
non-mutating extraction pre-pass `_detect_gc_preamble!` that recognises the
dead Julia GC/heap skeleton, proves it dead w.r.t. the return, and tells the
module walker which LLVM instructions to skip. `mem=:heap` opt-in mode threaded
`reversible_compile` → `extract_parsed_ir` → `_module_to_parsed_ir[_on_func]`;
normalised to `:auto` before `lower()` (no lowering change). `f1(x::Int8) =
let v=Int8[]; push!(v,x); v[1] end` compiles green under `mem=:heap` (identity
circuit, oracle over all 256 Int8). New `test/test_gps7_heap_m1.jl` 519/519.
Default `mem=:auto` path byte-identical — the recogniser returns inert unless
`mem===:heap`.

**Why:** the dynamic-memory frontier is the one coherent program left with
real user-facing value (worklog/072 handoff). M1 is the foundation: it clears
the inline-asm TLS wall that blocked every heap function, and ships the
dead-skeleton liveness proof that M2/M3 reuse.

**Architecture decision (3+1 synthesis).** Both architects agreed on
extraction-only recognition + a conservative bail-loud liveness proof. They
diverged on how to drop f1's dead `growend!` + capacity-diamond: Architect A —
erase the skeleton from the LLVM Module, run `dce,simplifycfg`; Architect B —
never mutate the Module, reuse the `sret_writes.suppressed` instruction-skip
idiom. Orchestrator chose **B**: A's erase-then-`simplifycfg` leaves the module
transiently malformed (an erased block terminator) and A under-specified that
fixup; mutating a live JIT'd module mid-extraction is fiddlier than it looks,
and CLAUDE.md §5/§10 lean against a second codegen stage. B reuses a *proven*
mechanism.

**The liveness proof (the correctness core).** `_recognise_skeleton` seeds
SKEL (all allocas, the `@ijl_gc_small_alloc`, the TLS asm, `llvm.memset` /
`growend!` calls, **and loads from non-SSA/global addresses** — "seed-5", see
gotchas) then forward-taint-closes (any instruction with a skeleton operand
joins SKEL). `_prove_skeleton_dead` runs 6 obligations, ALL required, ANY
failure → loud reject: P-recognise (closure ⊇ seeds), P-return (no SKEL
instruction is a `ret`-ancestor — the deadness proof), P-escape (every skeleton
store's address is skeleton-owned or a global), P-callee (every skeleton call
is TLS-asm / `gc_small_alloc` / `growend!` / benign `llvm.*`), P-opcode (no
skeleton instruction has an unmodelled side-effecting opcode —
atomicrmw/cmpxchg/invoke/callbr/fence), P-noload-into-live (no surviving load
reads skeleton-written memory — the worklog/029 `cond_pair` guard). Worst
realistic outcome is a spurious loud reject; a silent miscompile is
structurally precluded by the pervasive forward closure + all-allocas-seeded.

**Milestone-boundary finding.** f1's real `optimize=true` IR contains a
conditional `j_#_growend!` and a capacity diamond — which the consensus design
(`docs/design/heap_memory_consensus_design_2026-05-21.md`) nominally filed
under M3. Resolution: M1 owns the *fully-dead* diamond (the degenerate case —
the whole skeleton incl. the diamond is dead, surviving slice is straight-line);
M3 owns the *partially-live* case (skeleton dead but data live → re-rooting).
The boundary was clarified, not moved.

**Gotchas / lessons:**
- f1's GC skeleton SURVIVES `optimize=true` in full IR — but `ret` returns the
  argument `%x` directly, so the whole skeleton is dead w.r.t. the return.
- f4 (`Array{Int8}(undef,8)`, 8-elem) is FULLY DCE'd by LLVM SROA to a plain
  `alloca [3 x i64]` — no skeleton at all. The synthetic-alloca work is only
  needed for large / non-SROA-able arrays (M2 / TJ4's `undef,256`).
- `cond_pair` under `--check-bounds=yes` emits the skeleton but its element
  data pointer IS live into the return — an M2-shaped case; M1 correctly
  rejects it loud (P-return), never miscompiles it.
- `LLVM.operands(inst)` CRASHES (`error("Unknown value kind
  LLVMGlobalAliasValueKind")`) on the `GlobalAlias` operands pervading the
  heap skeleton (`store ptr @"jl_global#NNN.jit", ...`). heap.jl uses raw
  `LLVM.API.LLVMGetOperand` / `LLVMIsAInstruction` — no value identification.
  A landmine for any future code walking heap IR.
- The Julia 1.12 `Memory{T}` header data pointer is a module-scope global
  (`@"jl_global#NNN.jit"`), a *constant* not an SSA value — the header reads
  (`%memory_data`, `%.unbox`) have no tainted SSA operand, so the forward
  closure cannot reach them. Hence seed-5: loads from non-SSA addresses are
  seeded into SKEL structurally. Sound because a live such load is caught by
  P-return / P-noload (loads have no side effect — dropping a dead one is
  always safe; the only risk is dropping a *live* one).
- The substring `#_growend!##0#_growend!##1` appears inside an alloca's
  *value name* — the growend regex `^j_#_growend!##\d+_\d+$` must be matched
  against callee FUNCTION names only, never value names.
- f1's IR has THREE allocas (gcframe `[7 x ptr]`, growend-scratch `[9 x i64]`,
  sret_box `[2 x i64]`) — all skeleton. Don't assume "one alloca."
- `invoke`/`callbr` are NOT opcode `LLVMCall` — they slip past a naive
  `opcode==Call` callee check yet the taint closure reaches them (the closure
  only excludes Ret/Br/Switch/IndirectBr/Unreachable). P-opcode (added in
  orchestrator review) closes that gap.

**Rejected alternatives:** Architect A's Module-mutation + `dce,simplifycfg`
