## Session log — 2026-05-01 — Bennett-u71l / U161 close (CompileOptions struct)

**Shipped:** see git log around the next commit. Three changes:

1. **`Base.@kwdef struct CompileOptions`** added in `src/Bennett.jl` with
   all 9 fields (`optimize`, `max_loop_iterations`, `compact_calls`,
   `bit_width`, `add`, `mul`, `strategy`, `fold_constants`, `target`).
   `const _DEFAULT_COMPILE_OPTIONS = CompileOptions()` is the single
   source-of-truth for defaults.

2. **All three existing kwarg overloads** (Tuple, ParsedIR, Float64)
   now reference `_DEFAULT_COMPILE_OPTIONS.<field>` for their kwarg
   defaults instead of repeating literal defaults. The kwarg API surface
   is unchanged — every existing caller keeps working.

3. **Three new overloads** accept `opts::CompileOptions` positionally
   and forward each field as a kwarg:
   - `reversible_compile(f, arg_types::Type{<:Tuple}, opts)`
   - `reversible_compile(parsed::ParsedIR, opts)`
   - `reversible_compile(f, ::Type{Float64}[, ::Type{Float64}[, ::Type{Float64}]], opts)`
   Per-overload applicability enforced via `_check_field_at_default`:
   ParsedIR rejects non-default `optimize`/`bit_width`/`strategy`;
   Float64 rejects non-default `bit_width`. Same error class as the
   existing kwarg cross-rejection (`ArgumentError`).

**Why:** U161 catalogue item — three overloads carried divergent kwarg
surfaces with literal defaults duplicated in three places. With this
change, the struct is the canonical place defaults live, and users with
a config they want to reuse can pass `CompileOptions(...)` once.

**Gotchas / Lessons:**

- **No deprecation of the kwarg paths yet.** The catalogue suggested
  "deprecate kwarg-only paths" but that would emit warnings on every
  one of the 565 existing kwarg call sites — too noisy for one PR. Left
  the kwarg API in place; the struct is additive. Future deprecation
  can ride on top.
- **Struct field order matches the kwarg ordering** in the existing
  signatures, so each forwarded field reads in sync with the kwarg
  signature.
- **`_check_field_at_default` uses `==` against the default** so a
  user passing `CompileOptions(bit_width=0)` to ParsedIR is OK
  (default), but `bit_width=8` raises. This matches the kwarg-side
  intuition: "this option doesn't apply on this overload."
- The Float64 overload is variadic over `Type{Float64}...` (1, 2, or 3
  args). The CompileOptions wrapper provides explicit 1-/2-/3-arg
  methods rather than chasing the splat — three short methods are
  clearer than one method with index-walking. Same fan-out the kwarg
  side already does internally.

**Tests:** `Pkg.test()` clean — same baseline asserts pass; zero gate-
count drift; new smoke tests confirm the opts-form gates match the
kwarg-form gates exactly.

## Session log — 2026-05-01 — Bennett-s92x / U115 close (SretInfo struct + _collect_sret_writes decomposition)

**Shipped:** see git log around the next commit. Two refactors in
`src/extract/sret.jl`:

1. **`_detect_sret` ad-hoc NamedTuple → `struct SretInfo`.** The 7-field
   NamedTuple return became a documented struct with the same fields. All
   field accesses (`sret_info.n_elems`, `sret_info.elem_width`, ...) work
   unchanged. The NamedTuple-returning constructors at `module_walk.jl:67-68,
   98` and the field accesses at `sret.jl:125-129, 181, 186, 364-365` all
   pass through cleanly.

2. **`_collect_sret_writes` 173-line write-pattern soup → 4 per-pattern
   handlers + a clean dispatch loop.** Each `_try_handle_sret_*!` returns
   `Bool` ("did I claim this instruction?"); the main loop short-circuits
   on the first claim:
   - `_try_handle_sret_memcpy_reject!` — rejects optimize=false llvm.memcpy form
   - `_try_handle_sret_gep!` — tracks byte offset of sret-derived constant-offset GEPs
   - `_try_handle_sret_vector_store!` — handles SLP-emitted vector stores (Bennett-0c8o)
   - `_try_handle_sret_scalar_store!` — handles canonical scalar integer stores

   The collector body shrank from 173 lines to ~40, with the per-pattern
   logic moved into named functions whose docstrings explain what they
   handle. The return value is also promoted from a 4-field NamedTuple to
   a `struct SretWrites` for documentation parity with `SretInfo`.

All **419,228 tests pass** in 3m48s. Gate counts unchanged.

**Why:** U115 noted that `_collect_sret_writes` was a 173-line "special-case
soup" of inline `if opc == ...` arms with deeply-nested error checks, and
that `_detect_sret` returned an undocumented NamedTuple. Both made the file
hard to navigate: adding a new sret write pattern (e.g. `llvm.memset` for
`fill!`-like Julia idioms) required reading the whole loop to find the
right insertion point. Now each pattern is a separately-named function with
its own docstring; new patterns slot in as a new `_try_handle_sret_*!` plus
one line in the collector loop.

**No 3+1 needed:** the file was already in a bounded scope (378 LOC in
`src/extract/sret.jl` post-x3jc 2026-04-30); the decomposition is mechanical
extraction with no semantic change. New code = the SretInfo/SretWrites
structs (10 LOC each, just typed field declarations) + the per-pattern
function signatures (already implicit in the inline if-arms).

**Gotchas / Lessons:**

1. **`_try_handle_*!` "claim and return" pattern shines for LLVM walk loops.**
   Each handler decides "is this my instruction?" by checking opcode + operand
   shape, returns `false` cheaply if not. The main loop becomes a list of
   `handler(...) && continue` — strict precedence by source order. Adding a
   new pattern is one new function + one new line. Compare to a giant `if
   opc == X / elseif opc == Y / ...` chain where dispatch and body are
   entangled.

2. **The vector-store handler must run before the scalar-store handler.**
   Both target `LLVMStore`, but the vector handler checks `vt isa
   LLVM.VectorType` and the scalar handler asserts `vt isa LLVM.IntegerType`
   (failing loud on vector). Swap the order and every SLP-emitted vector
   sret store would crash with "non-integer value type". The dispatch loop's
   textual order encodes this precedence — kept the original precedence by
   listing vector before scalar.

3. **`_try_handle_sret_memcpy_reject!` returns false on success.** The
   memcpy-reject handler either errors loud (sret-targeting memcpy) or
   *doesn't claim the instruction* (other memcpy calls — the regular block
   walk handles them). It's the only handler that returns `false` after real
   work. Naming it `_check_*` was considered, but kept consistent with the
   `_try_handle_*` shape so the dispatch loop reads uniformly.

4. **`Union{Nothing, SretInfo}` return for `_detect_sret`.** Pre-refactor was
   `Union{Nothing, NamedTuple{...}}` — same shape, but the NamedTuple
   spelling was a wall of types in stack traces. `SretInfo` gives Julia a
   one-token type to display — small win but real.

**Rejected alternatives:**

- **Move SretInfo/SretWrites to `src/ir_types.jl`.** The other IR types
  live there for parametric eltype reasons. SretInfo and SretWrites are
  extractor-internal — no other module references them by name, and the
  LLVM.jl types in their fields (`LLVM.ArrayType`, `_LLVMRef`) wouldn't be
  appropriate in a Julia-native IR types file. Kept them in `sret.jl`.

- **Make `_try_handle_*!` return the byte offset / slot tuple.** Would
  enable a more "functional" composition. Rejected — the handlers mutate
  the dicts directly (slot_values, gep_byte, suppressed, pending_vec),
  and threading those through return values would just push the mutation
  to the dispatch loop.

**Filed (follow-ups):** none.

Remaining Section B P2 catalogue work: `vpch` (162 typed-error sites

**Why:** U56 noted the (4,8)/(8,8) helpers and the @eval-generated
(2,8)/(2,16)/(4,16)/(2,32) helpers had textually-identical bodies — the
hand-written variants pre-dated the parametric loop by one milestone (M1
vs M2a) and were never folded back. The shape set was implicitly hardcoded
twice: once in `_MUX_EXCH_STRATEGY` and once in the `@eval` loop list.
Adding a new shape required edits in both. This refactor establishes
`_MUX_SHAPES_NW` as the single source.

**No 3+1 needed:** pure consolidation, zero new code (the @eval loop and
the soft callees `soft_mux_*_4x8` / `soft_mux_*_8x8` already existed; the

wire-pack/unpack boilerplate that the @eval generates).

**Gotchas / Lessons:**

1. **`@eval`-time `getfield(@__MODULE__, sym)` for the dispatch dicts.**
   The dispatch tables can't reference the per-shape lowering functions by
   name in source (they're generated dynamically by the loop). Comprehension
   over `_MUX_SHAPES_NW` with `getfield(@__MODULE__, Symbol(:..., N, :x, W,
   :!))` resolves them at module-evaluation time after the @eval loop has
   defined them. Order matters: the @eval loop must precede the dispatch
   dicts in source (it does — line 509 vs line 600).

2. **`_MUX_EXCH_STRATEGY` keys are `(W, N)`, not `(N, W)`.** The shape list
   uses `(N, W)` order (matching the function-name suffix `_NxW`); the
   strategy dict is keyed on `(elem_w, n_elems)` order to match
   `alloca_info::Dict{Symbol, Tuple{Int,Int}}` and `_pick_alloca_strategy(shape::Tuple{Int,Int}, ...)`.
   The comprehension swaps the order: `(W, N) => Symbol(:mux_exch_, N, :x, W)`.
   Confused me for a minute on first read; tested with `sort(collect(keys(...)))`
   and confirmed the (8, 2) / (8, 4) / (8, 8) / (16, 2) / (16, 4) / (32, 2)
   set matches the pre-refactor dict.

3. **The hand-written load functions lived in `aggregate.jl`, not `memory.jl`.**
   Pre-vdlg this didn't matter — both were in the monolithic `lower.jl`.
   After the vdlg split, the load functions were grouped with `_lower_load_via_shadow!`
   in `aggregate.jl` and the store functions with the @eval generator in
   `memory.jl`. Folding everything into the @eval moves the load
   definitions across files — the `aggregate.jl` deletion site now has a
   pointer comment to where they live now.

4. **Hand-written load was missing M2d block-guard logic.** Bennett-cc0 M2d
   added block-predicate guards to the @eval-generated stores (so non-entry
   block stores route to `soft_mux_store_guarded_NxW`). The hand-written
   stores got the same treatment. The hand-written loads… didn't need it
   (loads are unconditional w.r.t. block predicate per the docstring at
   `_lower_load_via_shadow_checkpoint!`). The @eval load helper also has no
   guard logic. Equivalent, no change.

**Rejected alternatives:**

- **Keep the (4,8)/(8,8) hand-written variants for "explicitness."** Considered
  briefly — the early shapes might benefit from named visibility for
  documentation purposes. Rejected because the @eval-generated variants are
  exactly as inspectable (`@which _lower_store_via_mux_4x8!(args...)` lands
  on the @eval loop body, which is ~80 lines and well-commented). Documentation
  benefit is illusory; duplication cost is real.

- **Move the @eval loop to a separate file (`lowering/mux_exch_codegen.jl`).**
  Considered for clarity. Rejected because the @eval loop, the
  `_MUX_SHAPES_NW` source, and the three dispatch dicts form a tight
  coherent unit — splitting them across files would break the read order
  (you have to read the loop to understand what the dispatch comprehensions
  resolve to). Keep them adjacent in `memory.jl`.

**Filed (follow-ups):** none. Adding a new shape (e.g. `(8, 16)`) is now a
one-line addition to `_MUX_SHAPES_NW` plus the corresponding `soft_mux_*_NxW`
callees in `softmem.jl` (which are themselves parametric).

## Session log — 2026-05-01 — Bennett-ehoa / U43 close (LoweringCtx ::Any → concrete dict types)

**Shipped:** see git log around the next commit. Three `::Any` fields on
`LoweringCtx` (`preds`, `branch_info`, `block_order`) tightened to their
documented concrete dict types:

- `preds::Dict{Symbol,Vector{Symbol}}` — block label → predecessor labels
- `branch_info::Dict{Symbol,Tuple{Vector{Int},Symbol,Symbol}}` — block label → (cond_wires, true_label, false_label)
- `block_order::Dict{Symbol,Int}` — block label → topological order index

Pure type tightening — no new code, no semantics change, no constructor
signature change. All **419,228 tests pass** in 3m44s. Gate counts
unchanged.

**Why:** U43 description claimed "10-30% lowering speedup" from concretizing
these fields — every `_lower_inst!` dispatcher reads `ctx.preds`,
`ctx.branch_info`, `ctx.block_order`, and the `::Any` annotation forced
runtime dispatch on each field access (Julia's type inference can't see
through `::Any` for inlining). The performance claim is plausible but not
the primary driver — the bigger win is making `LoweringCtx` actually
*describe its contract*. The defensive `::Any` was added as "accept any
dict shape from caller" but no caller ever exercised the looseness:
`grep "preds = Dict\|branch_info = Dict"` shows every construction site
already uses the canonical types.

**No 3+1 needed:** per the user's 2026-05-01 directive ("LOC refactors
don't need 3+1 if no new code is written"). This is a 3-line annotation
edit with zero new code; the existing callers already pass the concretized
types.

**Gotchas / Lessons:**

1. **`ehoa` had a "two halves" history.** The bead description mentioned
   "16 fields, 4 constructors (7/12/13/14-arg) accreting M2a→M3a milestones"
   and a related `ParsedIR::memssa::Any` — but those were closed as
   drive-bys: Bennett-tbm6 (2026-04-27) removed the back-compat constructors
   (`src/lowering/types.jl:109-112` records the removal), and Bennett-by8j
   concretized `ParsedIR::memssa` to `Union{Nothing, MemSSAInfo}` by moving
   the type definition into ir_types.jl ahead of ParsedIR. The remaining
   v958-prerequisite work was just the 3 `::Any` fields on LoweringCtx —
   essentially a one-edit close once IROperand was no longer in the way.

2. **`v958` was NOT a hard prerequisite for ehoa.** I pencilled it in as a
   prerequisite ("operand spine type-stable first") in the v958 closing
   note, but reading the actual ehoa surface, the IROperand refactor and
   the LoweringCtx field types are independent. Either could have gone
   first. v958 has the larger blast radius (47 sites vs 3 fields), so doing
   it first reduced overall risk; ehoa would have been mechanically the
   same either way.

3. **The "::Any to accept any dict shape from caller" idiom is anti-pattern
   defence — it propagates type instability everywhere.** Better practice
   when you genuinely need a flexible dict shape: parameterise the struct
   (`LoweringCtx{P,B,O}`), or use a concrete `AbstractDict` supertype with
   the eltype contract pinned. `::Any` is a load-bearing claim that needs
   justification; if no caller actually exercises the flexibility, delete
   it.

## Session log — 2026-05-01 — Bennett-v958 / U68 close (IROperand tagged union → type hierarchy)

**Shipped:** see git log around the next commit. Replaced the `IROperand`
tagged-union struct (`kind::Symbol` discriminant, `name::Symbol` + `value::Int`
co-resident fields, four magic-named `:const` sentinels) with a proper Julia
abstract type hierarchy:

- `abstract type IROperand`
- `SSAOperand(name::Symbol)` — the SSA-variable leaf
- `ConstOperand(value::Int)` — the integer-constant leaf
- `OpaquePtrSentinel` / `PoisonLaneSentinel` / `ZeroAggSentinel` — empty
  singletons (`OPAQUE_PTR_SENTINEL`, `POISON_LANE`, `ZERO_AGG`)
- `PendingVecLane(lane::Int)` — sret-aggregation placeholder, the only
  sentinel with a payload

`resolve!` is now multi-method (one per leaf), with a catch-all
`resolve!(::IROperand, ...) = error(...)` fallback that fails loud per
CLAUDE.md §1 if any new sentinel reaches lowering without its own method.
`_ssa_operands` collapses via a `_ssa_names(::IROperand) = ()` helper that
specialises to `(op.name,)` for `SSAOperand` — every per-IR-type method
becomes a single splatted `Symbol[...]` literal.

All **419,228 tests pass** in 3m50s. Gate counts unchanged (verified at
i8/i16/i32 against the §6 baselines).

**Why:** Bennett-v958 / U68 was the largest remaining "core type change"
P2 from the catalogue. The tagged union was the spine — every operand
field in every IRInst was a concrete `IROperand` carrying both a Symbol
and an Int regardless of which one was meaningful, with 47 `.kind == :ssa`
/ `.kind == :const` discriminator sites scattered across `src/lowering/`,
`src/extract/`, and `src/qrom.jl`. The four magic-named sentinels
piggy-backed on the same struct (e.g. `IROperand(:const, :__opaque_ptr__,
0)` for the OPAQUE_PTR placeholder) — primitive obsession all the way down.

**3+1 protocol:** ran two parallel `Plan` proposers with the full 47-site
+ 4-sentinel survey. Both converged on the abstract-type + multi-dispatch
shape. They diverged on sentinel factoring: A proposed per-singleton
subtypes (`OpaquePtrSentinel`, `PoisonLaneSentinel`, etc.); B proposed a
single `SentinelOperand{kind::Symbol}` umbrella to reduce method-table
pressure. Picked A — B's design reintroduces the `kind::Symbol`
discriminant inside the sentinel bucket, defeating the point of the refactor;
A's "method-table bloat" is illusory because `_ssa_names(::IROperand) = ()`
covers all 4 sentinels with one zero-cost fallback.

**Gotchas / Lessons:**

1. **`Vector{SSAOperand}` is NOT `Vector{IROperand}` in Julia** (parametric
   invariance for mutable containers). Pre-refactor, `[ssa(:a), ssa(:b)]`
   was inferred as `Vector{IROperand}` because `IROperand` was concrete.
   Post-refactor, the same expression infers as `Vector{SSAOperand}` and
   silently breaks every `IRCall(_, _, args, ...)` / `IRPhi(_, _, incoming,
   ...)` / `IRSwitch(_, _, _, cases)` call site that doesn't manually
   annotate the eltype. **Cheap systemic fix:** widen the inner constructors
   to accept `AbstractVector{<:IROperand}` (and `AbstractVector{<:Tuple{<:IROperand,
   Symbol}}`) and `convert(Vector{IROperand}, args)` internally. One
   constructor edit per IRInst beats hunting down ~50 call sites. **Idiom
   for future abstract-type-eltype refactors:** widen at the constructor,
   not at the call site.

2. **Static-inspection tests grep field names too.** `test_store_alloca_extract.jl`,
   `test_liveness.jl`, `test_u21m_switch_phi_patching.jl`, `test_0c8o_vector_sret.jl`
   all had `.kind == :ssa` / `.kind == :const && .name === :__X__` checks
   that are no longer valid syntax (the field is gone). Same fragility class
   as the vdlg / x3jc / zpj7 path-grep tests. Cheap rewrite via `isa`
   (`.kind == :const && .value == 10` → `isa Bennett.ConstOperand && .value == 10`).

3. **`Vector{IROperand}` field type stays — only the constructor widens.**
   Storing as the abstract eltype `Vector{IROperand}` is the canonical form
   (a phi can have mixed `(SSAOperand, ConstOperand)` incoming edges, and
   without abstract storage the dict can't hold both). The constructor
   coerces the caller's narrower vector once.

4. **`OPAQUE_PTR_SENTINEL` was exported and tested** — kept the name as a
   `const OPAQUE_PTR_SENTINEL = OpaquePtrSentinel()` binding so
   `test_ibz5_opaque_ptr_sentinel.jl`'s import line still resolves. The
   T2 test ("any OpaquePtrSentinel instance trips the tripwire") is now
   *stronger* than pre-refactor: pre-v958 it relied on the magic name
   `:__opaque_ptr__`; post-v958 it dispatches on the type itself, so even a
   freshly-constructed (non-canonical) `OpaquePtrSentinel()` trips it.

5. **`_IR_OPERAND_KINDS` validation tuple deleted** — was `(:ssa, :const)`,
   used by the old IROperand inner constructor to validate `kind`. The new
   types validate themselves (you can only construct `SSAOperand(::Symbol)`
   with a Symbol; the type system replaces the runtime tuple membership
   check). `test_k7al_ir_constructor_asserts.jl` "validation tables exist"
   testset updated to drop the OPERAND_KINDS reference; "IROperand: kind
   validation" testset rewritten to verify the type hierarchy
   (`isa SSAOperand`, `isa ConstOperand`, etc.) and that the abstract
   `IROperand()` construction throws `MethodError`.

6. **Single atomic commit, not phased.** A phased migration would require
   `IROperand` to be both an abstract type and a concrete struct
   simultaneously (rename one to dodge the conflict), which doubles the
   diff and leaves the codebase in a type-unstable intermediate state.
   The codemod is mechanical and the 47-site grep bounds the scope exactly.
   Net delta: ~+30 LOC in `ir_types.jl`, ~−40 LOC in `lowering/operand.jl`
   (the 16 `_ssa_operands` methods collapsed to one-liners), ~0 in the
   other 10 src files (mostly `.kind == X` → `isa Y`).

**Rejected alternatives:**

- **`SentinelOperand{kind::Symbol}` umbrella** (proposer B's design).
  Reintroduces the `kind::Symbol` discriminant inside the sentinel bucket
  — primitive obsession comes back, just within a smaller scope. Per-singleton
  subtypes are the cleaner factoring and the "method table bloat" cost is
  zero because the `_ssa_names(::IROperand) = ()` fallback covers all
  sentinels.

- **Reflective `_ssa_operands` via `fieldnames(typeof(inst))`.** Container
  fields (`IRCall.args::Vector{IROperand}`,
  `IRPhi.incoming::Vector{Tuple{IROperand,Symbol}}`,
  `IRSwitch.cases::Vector{Tuple{IROperand,Symbol}}`) break the naïve
  filter — fixing them puts the special cases back, just hidden behind
  `eltype` introspection. Type-stable explicit per-method dispatch on
  `IRBinOp`/`IRSelect`/etc. is what the hot `compute_ssa_liveness` loop
  benefits from.

- **Phased migration.** See gotcha #6.
