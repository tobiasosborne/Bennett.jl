**Empirical accuracy:**
- 100k random sweep across `[1e-22, 1e22]` (5 magnitude buckets, 3 seeds):
  sin max ULP **= 1**, cos max ULP **= 0**. **0 fails > 2 ULP** in any
  bucket. Bit-exact vs `Base.sin` for 99%+ of samples; ≤1 ULP on the
  rest.
- 1076-input subnormal-input sin sweep (every binade 2^-1075 to 2^0):
  **max ULP = 0** (bit-exact, sin(x) = x in tiny-arg regime is preserved
  exactly).
- Specials: NaN propagates with quiet-bit, ±Inf → NaN, sin(±0) = ±0
  (sign-preserving), cos(±0) = +1.0 bit-exact.

**Gate budget:**
- `soft_sin` compiles to **~11M reversible gates** (1.6M NOT, 7.1M CNOT,
  2.3M Toffoli). Compile time ~45s on the developer machine.
- `soft_cos` similar magnitude (a bit smaller; cos kernel has fewer ops).
- Each soft_f* call is itself thousands of gates, so the cost
  is dominated by paynehanek's three 64×64→128 multiplies + Cody-Waite
  ext's three-level cancellation cascade.

**Why:** The next P3 transcendental opcode after `Bennett-582` (log family)
and `Bennett-emv` (pow). User's directive: "full best practices solution
only" — meaning full Payne-Hanek argument reduction, not the limited-range
Cody-Waite-only port. Required for IEEE 754 conformance over the full
Float64 range and for ≤2 ULP vs `Base.sin` / `Base.cos` (which Julia
implements via `paynehanek` for `|x| > 2^28`).

**Algorithm — port of musl `sin.c` / `cos.c` / `__sin.c` / `__cos.c` /
`__rem_pio2.c` / `__rem_pio2_large.c`** (FreeBSD/SunPro 1993, BSD-licensed,
already shipping in glibc/musl/FreeBSD/Julia-via-openlibm):

  1. **Tiny-arg fast path.** sin: `|x| < sqrt(eps) ≈ 2^-26.5` → return x
     bit-exact (sin(x) ≈ x). cos: `|x| < 2^-27 · √2` → return 1.0.
  2. **Small-arg path** (`|x| < π/4`). Direct kernel call.
  3. **Medium-arg path** (`|x| < 2^28 · π/2 ≈ 8.4e8`). Cody-Waite reduction
     splits π/2 into a 33-bit `pio2_1 = 1.57079632673412561417e+00` plus a
     53-bit tail `pio2_1t = 6.07710050650619224932e-11`. For `|x|` up to
     `4·π/2`, dispatch by hardcoded constant `n ∈ {±1, ±2, ±3, ±4}`. For
     larger medium args, compute `fn = round(x · 2/π)` via the
     `x·invpio2 + toint − toint` rounding trick, then iteratively refine
     with three-level (pio2_1, pio2_2, pio2_3) Cody-Waite splits when
     cancellation eats precision (`ex - ey > 16` then `> 49` thresholds).
  4. **Huge-arg path** (`|x| ≥ 2^28 · π/2`). Payne-Hanek multi-precision
     reduction: extract the 53-bit mantissa `X` and exponent `k`, look up
     a 192-bit window of `2/π` from the 19-entry `INV_2PI` table at offset
     `idx = k >> 6` with `shift = k mod 64`, compute `w = X · 2/π_window`
     in 256-bit precision, isolate the fractional part `f`, derive
     quadrant `q = round(w / 2^125) mod 4`, and convert `f` (as Int128
     fixed-point) to a `(hi, lo)` Float64 pair via `fromfraction`.
     Multiply the result by `π/2` (split as `pio2_hi + pio2_lo`) to land
     in `[-π/4, π/4]`.
  5. **Quadrant select.** With `n & 3` and the reduced `(yhi, ylo)`:
     - sin: 0 → +sin_kernel(y), 1 → +cos_kernel(y),
            2 → -sin_kernel(y), 3 → -cos_kernel(y).
     - cos: 0 → +cos_kernel(y), 1 → -sin_kernel(y),
            2 → -cos_kernel(y), 3 → +sin_kernel(y).
  6. **Kernels.**
     - `sin_kernel(y_hi, y_lo)` — degree-13 polynomial in y_hi:
       `y + y·z·(S1 + z·r)` with tail correction `−((z·(0.5·y_lo − v·r) −
       y_lo) − v·S1)` where `z = y_hi²`, `v = z·y_hi`,
       `r = S2 + z·(S3 + z·S4) + z·z²·(S5 + z·S6)`.
     - `cos_kernel(y_hi, y_lo)` — degree-14 polynomial:
       `(1 − z/2) + (((1 − (1 − z/2)) − z/2) + (z·r − y_hi·y_lo))` with
       `r = z·(C1 + z·(C2 + z·C3)) + z²·z²·(C4 + z·(C5 + z·C6))`.
  7. **Special cases** (last-write-wins ifelse cascade):
     - sin(NaN) = NaN (propagate, force quiet-bit)
     - sin(±Inf) = NaN (= INDEF)
     - sin(±0) = ±0 (sign-preserving)
     - cos(NaN) = NaN
     - cos(±Inf) = NaN
     - cos(±0) = +1.0

**Bit-exactness target:** `≤2 ULP vs Base.sin / Base.cos` over the full
Float64 range. Bit-exact vs `ccall(:sin/cos, ...)` on Linux (system libm
= musl/glibc). This matches the soft_pow / Base.^ pattern (Bennett-emv):
ship the musl-tracking variant for raw `.ll`/`.bc` ingest; if a
Bennett-jexo-style Julia-tracking variant is needed later (for
`Base.sin(::SoftFloat) → SoftFloat(soft_sin_julia(x.bits))`), file a
follow-up bead.

**Why musl-tracking, not Julia-tracking up front:**
Julia's `cody_waite_ext_pio2` uses `muladd(-fn, pio2_1, x)` which Bennett-
jexo (2026-05-02) demonstrated does NOT contract to FMA in `@noinline`
contexts — it lowers to two roundings (`fmul contract; fadd contract`).
musl uses two separate ops (`r = x - fn*pio2_1; w = fn*pio2_1t`), so
matching musl's ops automatically matches Julia's @noinline rounding. In
@inline contexts the contraction is compiler-discretionary and bit-
exactness across compilers is not guaranteed; the empirical answer
(matched in Bennett-jexo) is that two-rounding wins. Hence: port musl,
expect bit-exactness vs `ccall(:sin)`, ≤2 ULP vs `Base.sin`.

**Bennett compile-to-gates constraints (recap from prior closes):**
- All arithmetic via `soft_f*` primitives (no native `*`/`+`/`fma` on
  Float64). The `_sf_widemul_u64_to_128` and `_add128` helpers in
  `softfloat_common.jl` are already there for the Payne-Hanek 64×64→128
  multiplies.
- Tuple-returning helpers must be destructured at the call site; NOT
  used inline within a single statement (Bennett-emv gotcha).
- Lookup tables stride-1 only (Bennett-emv gotcha — `INV_2PI[idx+0..2]`
  is fine).
- All control flow branchless via `ifelse` cascades.
- @inline on every helper so the IR walker sees a flat body.

**INV_2PI table** — 19 × UInt64, computed via:
```julia
setprecision(BigFloat, 4096); I = 0.5/big(pi)
for i = 1:19;  I *= 0x1p64; k = trunc(UInt64, I); println("0x", string(k, base=16, pad=16)); I -= k; end
```
Hardcoded verbatim from `base/special/rem_pio2.jl` (Julia's port, derived
from FDLIBM `__ieee754_rem_pio2`).

**DS1..DS6, DC1..DC6 polynomial coefficients** — verbatim from FreeBSD
`k_sin.c` / `k_cos.c` (Sun 1993). Identical between Julia/musl/FreeBSD/
glibc — they all carry the same Sun coefficients.

**Gotchas / Lessons:**

1. **i128 constants from native UInt128 are rejected by Bennett's IR
   walker (Bennett-l9cl / U09).** Bennett-yys3 found that UInt128
   *operations* don't emit compiler-rt calls in Julia 1.12, but they
   DO emit i128 *constants* for shift amounts (e.g. `shl i128 %x, 64`)
   and for any literal i128 input. The IR walker truncates such
   constants to Int64 silently (`IROperand.value` is Int64) and errors
   out via `_const_int_as_int`. Effect: any function destined for
   `reversible_compile` cannot use native UInt128 / Int128 — even though
   the operations themselves work fine in regular Julia. The first
   paynehanek port was a clean 20-line UInt128 transcription matching
   Julia's source verbatim and bit-exact in pure-Julia execution; it
   crashed compile with `ConstantInt with width 128 bits` from the
   `(w << 2)` shift. The fix was a hand-rolled (hi, lo) UInt64-pair
   representation throughout, using the existing
   `_sf_widemul_u64_to_128` / `_add128` / `_sub128` / `_neg128` helpers
   from `softfloat_common.jl` plus new `_rp_shl128` / `_rp_shr128` /
   `_rp_clz64` / `_rp_clz128` locals. Doubled the LOC of paynehanek
   and fromfraction; correctness contract pinned by direct comparison
   against the Julia-source fromfraction (200 random Int128 samples,
   0 fails after the m1-shift fix). Update Bennett-yys3 docstring or
   add a NB to softfloat_common.jl about the constant-vs-operation
   asymmetry.

2. **paynehanek's INV_2PI window indexing is a footgun for
   `idx_signed = -1`.** Julia source: `INV_2PI[idx+1]` (1-based), so
   for `idx = -1` the access is `INV_2PI[0]` → OOB → substitute 0.
   With our 0-based `_inv_2pi_lookup(i) = INV_2PI[i+1]`, the right
   mapping is `_inv_2pi_lookup(idx_signed + offset)` for offsets
   {0, 1, 2, 3} matching Julia's INV_2PI[idx+{1, 2, 3, 4}]. First
   pass had an off-by-one (`idx_signed - 1` for the first slot)
   that produced *plausibly-shaped* wrong outputs across the whole
   huge-arg regime: results in [-1, 1] but uncorrelated with the
   true value. The branchless dispatch evaluates paynehanek even for
   tiny inputs where `idx_signed` is deeply negative (e.g. -17 for
   ±0.0); ALL four lookup indices must be substituted with 0 when
   negative, AND the lookup result substituted with 0, otherwise
   LLVM emits `unreachable` for the @inbounds OOB access.

3. **`fromfraction`'s m1 shift simplification.** Julia source:
   ```julia
   m1 = ((x >> (n1-26)) % UInt64) << 27       # 26 bits at positions 27..52
   x2 = x - (UInt128(m1) << (n1-53))
   ```
   The naïve (hi, lo) port uses `m1_orig` (= m1 without the << 27
   shift; just 26 bits at low positions) and then `m1_orig << (n1-53)`
   — but that places the bits at (n1-53)..(n1-28), NOT at (n1-26)..
   (n1-1) where they belong. Correct shift is `m1_orig << (n1-26)`,
   collapsing Julia's two shifts. First-pass bug produced z2 values
   roughly 2^25× too large (~0.25 instead of ~5e-9) — large enough to
   show up as catastrophic ULP error on the very first sample, small
   enough that a lazy spot-check on simple inputs (z1 alone) might
   miss. Pinned by 200-sample direct fromfraction-vs-Julia test.

4. **Branchless `>> 64` is undefined in LLVM, must be guarded.**
   `raw_x >> (64 - shift)` becomes `>> 64` when `shift == 0`. Julia's
   own `>>` happens to return 0 for shift ≥ 64, but LLVM's `lshr i64,
   64` is poison. Ifelse-guard each such site: substitute `0` for
   the discarded-branch path. Same in `_rp_shl128` / `_rp_shr128` for
   the cross-half terms.

5. **Tiny inputs (±0, subnormals) flow through paynehanek in the
   branchless dispatch.** The outer ifelse selects the small/tiny
   path's result, but paynehanek itself runs and must not crash. All
   internal shifts and lookups must tolerate `idx_signed` deeply
   negative. The fix is index-clamping at every lookup site.

6. **`UInt32(n)` for `n::Int32` throws InexactError on negatives.**
   `cody_waite_ext` returns `n_corr::Int32` which can be negative
   (cody_waite for x < 0). Compute `n & 3` for the quadrant select
   via `(n % UInt32) & UInt32(0x3)` — two's-complement reinterpret —
   not `UInt32(n) & UInt32(0x3)`. First pass crashed on `sin(-0.0)`
   with `convert(UInt32, -1)`.

7. **musl __sin's iy=1 vs iy=0 paths converge when ylo=0.** The
   Julia kernel `sin_kernel(y::Float64)` is the iy=0 case (no tail);
   `sin_kernel(y::DoubleFloat64)` would be the iy=1 case. We use the
   iy=1 form throughout (with ylo=0 for the small-arg path) — the
   formula `yhi - ((z*(0.5*ylo - v*r) - ylo) - v*S1)` collapses to
   `yhi + v*(S1 + z*r)` when ylo=0. Single code path = simpler
   maintenance, no measurable accuracy loss (verified by the smoke
   testset getting 0 ULP for small args).
