## Session log — 2026-05-14 — Bennett-k31q close: NOT a test-ordering bug — it's `--check-bounds`

**Shipped:** `test/test_t0_preprocessing.jl` — capture the full extractor
error message in `skipped` instead of truncating at 80 chars; truncate
display-only (char-safe `first`); print the active `--check-bounds` mode;
header warning about check-bounds sensitivity. See git log around this
session's commit. `_handle_memset_arm` deliberately left untouched.

**Root cause (the real one).** `Bennett-k31q` was framed — and twice
mis-handled — as a "test-ordering dependency": `test_t0_preprocessing.jl`
passes standalone but fails in full `Pkg.test()`. It is **not** an ordering
dependency and shares **no** mutable global state. The driver is
`--check-bounds`. `Pkg.test()` hard-codes `--check-bounds=yes`; a bare
`julia --project test/...` uses the default. Under `--check-bounds=yes`,
`cond_pair` (`x -> [x,-x][1+Int(x<0)]`) can no longer stack-allocate its
array literal — Julia emits a GC frame, which brings a **volatile**
`llvm.memset`, an **inline-asm** thread-pointer read (`movq %fs:0`), and an
`ijl_gc_small_alloc` **external call**. All three are correctly fail-loud
rejected by Bennett (out of scope, not gaps). Under the default flag the
same function compiles to a plain `alloca [18 x i8]` — no memset, no GC
frame, extracts clean. Confirmed: the test fails standalone too the moment
you pass `--check-bounds=yes`, and passes under `Pkg.test()` after the fix.

**The actual bug.** `test_t0_preprocessing.jl` already has an allowlist for
exactly these benign fail-loud skips — it contains `"volatile"`,
`"inline-asm"`, `"call ptr asm"`, `"atomic store"`. But the catch block
captured `sprint(showerror, e)[1:min(80,end)]`, and Bennett's verbose
fail-loud messages put the diagnostic keyword well past char 80 (the
`cond_pair` message is 399 chars; `"volatile"` sits at ~char 130). So the
allowlist grep never saw the keyword and every long message looked
"unexpected". Fix: capture the **full** message; truncate display-only
(char-safe `first(msg, 160)`, so the multibyte `—` in Bennett error
messages never splits on a byte boundary).

**Rejected alternative.** Subagent A's recommended fix was to reorder the
`_handle_memset_arm` predicate cascade so the `c==0` silent-drop fires
before the volatile check. Rejected: (1) it would be a core-file change
(3+1) for no real benefit — `cond_pair` hits inline-asm two instructions
after the memset, so "fixing" the memset just moves the failure to the next
construct; (2) `_handle_memset_arm` is *correct* — a volatile memset
genuinely cannot be honoured, and any Julia function carrying a volatile
GC-frame memset also carries the inline-asm + GC alloc that come with a GC
frame, so it is un-compilable regardless. The bug was never in the extractor.

**Gotchas / Lessons.**
* `test_t0_preprocessing.jl` is `--check-bounds`-SENSITIVE. It MUST be
  validated under `--check-bounds=yes` (= the `Pkg.test()` configuration). A
  standalone `julia --project test/...` run uses the default and will not
  reproduce the check-bounds=yes-only skips — this is exactly what got k31q
  wrongly closed on 2026-05-13. The test now prints its active check-bounds
  mode and warns when it is not `=yes`.
* Truncating a captured error message *before* keyword-matching it is a trap:
  the truncation silently neuters the allowlist. Match on the full string;
  truncate only for human display, and char-safe.
* Two Sonnet subagents (general-purpose, parallel, blind) both independently
  identified `--check-bounds` as the root cause. Subagent A confirmed it with
  real `--check-bounds=yes` IR; subagent B's specific-mechanism guesses
  (ConstantExpr byte count / atomic memset variant) were wrong because it
  never ran the check-bounds=yes IR — but its root-cause *ranking* was right.
  Lesson: a subagent that runs the actual reproduction beats one reasoning
  from code alone — and the orchestrator must still verify (subagent A's
  *recommended fix* was wrong even though its *diagnosis* was right).
