**Shipped:** see git log around the s1zl close commit. First close in
Tier C1 (trig completion) of the Enzyme parity north-star filed
2026-05-03. `soft_tan` is a faithful port of musl `tan.c` / `__tan.c`,
reusing `_rp_rem_pio2` from `fsin.jl` as-is for argument reduction.
`llvm.tan.f64` dispatches directly to `soft_tan` in
`src/extract/instructions.jl`, mirroring the Bennett-3mo (sin/cos)
arms; f32 rejected per CLAUDE.md §13. `_CALLEES_FP_TRANS` extended

**Empirical accuracy:** 100k random samples × 5 magnitude buckets ×
3 seeds (= 500k samples up to |x| ≤ 1e22). **Max ULP = 1, zero fails
> 2 ULP.** Bit-exact (0 ULP) on tan(π/4), tan(π/2 ± δ) for the swept
δ-grid, and the 1076-input subnormal-INPUT binade sweep.

**Why:** Tier C1 was pinned 2026-05-03 in
`Bennett-Enzyme-Parity-NorthStar.md` (the prior session's research
artifact). `tan` is the natural first close in C1 because it's the
only remaining trig function that doesn't need a new argument-
reduction kernel — `_rp_rem_pio2` from `fsin.jl` works as-is
(sin/cos/tan all share parity-aware n·π/2 reduction; the only
difference is the kernel that consumes the reduced (yhi, ylo) pair).
asin/acos/atan have bounded argument range and no Cody-Waite /
Payne-Hanek path; sinh/cosh/tanh need `soft_exp` plumbing. tan was
strictly the cheapest first move.

**Gotchas / lessons:**

1. **The "big" branch is INSIDE the kernel, not the dispatcher.** musl's
   `__tan.c` folds when `|y| ≥ 0.6744` (`hx >= 0x3FE59428`), AFTER
   range reduction. Easy to misplace at the soft_tan dispatcher level.
   The fold-rationale is that the polynomial is centered at 0 and
   argument range after `_rp_rem_pio2` is up to π/4 ≈ 0.7854, past the
   precision sweet-spot. Folding to `π/4 - |y|` keeps the polynomial
   argument near 0. We branchlessly compute BOTH the small-arm and
   big-arm polynomial paths and ifelse-select — 2× polynomial cost,
   constant dispatch.

2. **The odd-arm `-1/(x+r)` precision trick is non-trivial.** musl's
   `__tan` uses SET_LOW_WORD (zero low 32 mantissa bits) twice: once
   on `w_final` to get `z₂`, once on `a = -1/w_final` to get `a₀`.
   Then `result = a₀ + a·(1 + a₀·z₂ + a₀·v₃)` where
   `v₃ = r_final - (z₂ - x)` so that `z₂ + v₃ = r_final + x` exactly.
   This is essentially a Newton-Raphson refinement step on the
   reciprocal. Bennett's contract is ≤2 ULP, so the simple
   `-1/(x+r)` form would suffice — but the full trick costs ~5 extra
   ops and gives bit-exact match with musl. Kept the full version.

3. **`_RP_PIO4_BITS` from `fsin.jl` IS the high-part `pio4`** musl
   __tan.c needs (`0x3FE921FB54442D18` = 0.7853981633974483). Reused
   directly; only the round-off term `pio4lo` (3.06e-17) is new and
   went into ftan.jl as `_TAN_PIO4LO`. Pattern-match for other Tier C1
   ports: scan fsin.jl's constants table before adding new ones.

4. **Subnormal-output sweep collapses to subnormal-INPUT sweep for tan.**
   tan(subnormal) = subnormal because tan(x) ≈ x in the tiny-arg
   regime, and the `is_tiny` path (`|x| < 2^-27`, `xhp < 0x3E400000`)
   returns `a` bit-exact. Subnormal outputs near k·π would require
   input within ~2^-1022 of k·π — unreachable from a single Float64
   (spacing near π is ~2^-51). Same analysis as the cos near-π/2
   cancellation note in `test_softfsin.jl`. CLAUDE.md §13 contract
   met by the binade sweep `2^-1075 → 2^0` in test_softftan.jl.

5. **n_odd parity dispatch is the only difference from sin/cos at the
   dispatcher level.** sin/cos use `n & 3` (4-way quadrant); tan uses
   `n & 1` (2-way: even → tan, odd → -cot). Both go through the same
   `_rp_rem_pio2` and read `n` as `Int32`. The kernel consumes `odd`
   as a `Bool`, which is just `(n & Int32(1)) != Int32(0)`.

**Rejected alternatives:**

- **CORE-MATH 1-ULP correctly-rounded tan**: ~600 LOC vs musl's
  ~150 LOC, fresh argument-reduction path that doesn't reuse
  fsin.jl's `_rp_rem_pio2`. Higher LOC for marginal accuracy gain
  (1-ULP guaranteed vs Bennett's empirical max-1-ULP, contract 2-ULP)
  on a function that already meets/beats contract on 500k samples.
- **Arm Optimized Routines**: doesn't ship double-precision tan in
  the `arm-optimized-routines/math` directory (only sin/cos/exp/log/
  pow). Same gap as for sqrt — Bennett-1pb / Bennett-ux2 found this.
- **Julia stdlib `Base.tan`**: lowers to libm `tan` via ccall. Same
  source as musl underneath; porting via openlibm wrapper would add
  surface area and the `__ieee754_*` indirection without changing
  the underlying algorithm.
- **Simplified odd-arm `-1/(x+r)`**: ~2-ULP accuracy, would meet
  Bennett's contract. Rejected for the bit-exact match with musl
  output — cost is small (~5 ops in a circuit that already runs to
  many MGate).

**Next agent starts here:** Tier C1 has 10 siblings remaining. Cheapest
order:
1. **`atan` / `asin` / `acos`** (file as 3 separate beads). Bounded
   argument range — no Cody-Waite or Payne-Hanek. asin/acos reduce
   to atan via identities. atan needs its own polynomial + range
   split (|x|<1 vs |x|≥1). Probably ~300 LOC each, no shared
   infrastructure with the sin/cos/tan family.
2. **`atan2`** — 2-arg variant; quadrant dispatch around `atan(y/x)`.
   File separately because the 2-arg dispatch needs its own LLVM
   ingest path (atan2 is a libm call, not an `llvm.atan2.*` intrinsic
   in LLVM ≤17 — verify before filing).
3. **`tanh`** — needs `soft_exp` (already done in `fexp.jl`); use
   the `tanh(x) = (e^{2x} - 1) / (e^{2x} + 1)` form with the standard
   `|x|>20 → ±1` cutoff. ~150 LOC.
4. **`sinh` / `cosh`** — both need `soft_exp` plus careful handling
   of overflow. cosh(x) = (e^x + e^{-x})/2, sinh(x) = (e^x - e^{-x})/2.
   Common form-of-`expm1`-trick for sinh near zero — possibly defer
   sinh until expm1 (Tier C2) is available.
5. **`asinh` / `acosh` / `atanh`** — all reduce to logs.
   asinh(x) = log(x + √(x² + 1)), acosh(x) = log(x + √(x² - 1)),
   atanh(x) = 0.5·log((1+x)/(1-x)). Need `soft_log` (done) and
   `soft_fsqrt` (done). Watch for cancellation in atanh near 0.

After C1: Tier C2 starts with `expm1` (precision-critical, can't be
faked via `exp(x) - 1`) and `log1p` (same). Both have well-known
musl ports.
