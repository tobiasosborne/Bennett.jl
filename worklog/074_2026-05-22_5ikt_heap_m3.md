## Session log — 2026-05-22 — Bennett-s0tn — fail-loud loop-overflow detection (3+1 implementer)

P1 bug: `lower_loop!` unrolls a data-dependent loop to exactly K =
`max_loop_iterations` MUX-guarded body copies and STOPS — no check that
the loop's exit condition holds after iteration K. Undersized-K inputs
got a silent wrong answer (CLAUDE.md §1 violation). Implemented the 3+1
consensus design (`docs/design/s0tn_loop_overflow_consensus_design_2026-05-22.md`).

**Mechanism.** A fourth wire-partition class `loop_check`: `lower_loop!`
emits a forward-pass convergence wire `conv_w`; `_bennett_default` copies
it out parallel to the output copy-out into a frozen `conv_copy` wire that
survives the reverse pass; `simulate` / `verify_reversibility` error loud
when `conv_copy` reads 0. New struct `LoopGuard{wire, header_label, K}`.

### DEVIATION FROM CONSENSUS DESIGN (load-bearing — reported to orchestrator)

The design claimed iteration K's `exit_cond_wire` already holds the
convergence bit and a CNOT from it suffices. **It does NOT.**
`exit_cond_wire` from iteration K is the exit condition evaluated at the
START of iteration K — on the post-(K-1) loop-carried state. The unrolled
K-iteration circuit is CORRECT iff the loop would exit AFTER iteration K,
i.e. the condition is true on the POST-K state.

Witness: `countdown(4)` with K=4 runs n: 4→3→2→1→0. Iteration 4's start
condition is `n=1 > 0` = "not done", yet the loop genuinely converged in
exactly 4 iterations. Using iteration-K `exit_cond_wire` made
`simulate(c, Int8(4))` throw a false overflow.

**Fix:** after the K-iteration `for`, run ONE more *check-only*
evaluation — re-lower the header-body instructions through `_lower_inst!`
and recompute the header's exit condition on the post-K phi-frozen `vw`
values. No body blocks, no latch, no MUX — a (K+1)-th condition check.
Its result IS the true convergence bit. Built a fresh `conv_ctx`
`LoweringCtx` with iteration-local empty preds/branch_info dicts for this
pass. This is a small, local change to *which* wire feeds the convergence
CNOT — the rest of the design (copy-out, partition class, simulate check,
thread-through) stands as written.

### Gotchas / learnings

- **`countdown` is the wrong RED probe under `optimize=true`.** LLVM
  recognises `while n>0; n-=1 end` as the closed form `smax(x,0)` and
  eliminates the loop entirely — no back-edge, no LoopGuard. ALL loop
  tests must pass `optimize=false` (CLAUDE.md §5). Collatz is the
  exception: genuinely non-closed-form, its loop survives `optimize=true`
  — and it MUST stay `optimize=true` because under `optimize=false` the
  `&&` short-circuit emits an `__unreachable__` block that
  `_collect_loop_body_blocks` cannot resolve (pre-existing limitation,
  unrelated to s0tn).
- **The Bennett-reverse trap (design §"trap") is real.** A naive
  end-of-`simulate` check on `conv_w` is useless — the reverse pass
  uncomputes every non-output ancilla to 0. The copy-out-parallel-to-
  output-copy-out is the only thing that survives.
- **`verify_reversibility` on a finite-K loop circuit cannot pass random
  sweeps.** It randomises all 8 input bits; for `countdown` with K=20 a
  random input ≥ 21 genuinely overflows, and the new guard correctly
  throws. This is NOT a regression — it is the fix working. Had to
  rewrite `test_httg_loop_multiblock.jl` T1 (`_u05_accumulator`, K=6):
  its `verify_reversibility(c; n_tests=16)` was implicitly relying on the
  silent-wrong-answer bug. Replaced with explicit in-range `simulate`
  checks (which run the same ancilla/input-preservation assertions).
  Collatz self-caps at `steps<20` so a full random sweep is still safe
  there.
- **`exit_cond_wire` Julia for-body scoping** (design's CRITICAL note):
  after switching to the post-loop check-only pass, the hoist became
  unnecessary — `exit_cond_wire` is no longer read after the `for`. Left
  the in-body usage as-is.
- **`LoopGuard` lives in `src/gates.jl`, not `lowering/types.jl`.**
  `gates.jl` (include line 8) defines `ReversibleCircuit` whose
  `Vector{LoopGuard}` field must resolve at struct-definition time;
  `lowering/types.jl` loads much later (via `lower.jl`, line 32).
- **`call.jl` hardcoded `max_loop_iterations=64`** — confirmed smell.
  The loop guard works (K=64 appears in the overflow message), but the
  hardcode means a callee loop deeper than 64 silently can't be reached
  by a larger caller bound. Worth a follow-up bead.

**Why:** M3 is the third step of the dynamic-memory arc — M1 dropped a
wholly-dead heap collection, M2 re-rooted `Array{T}(undef,const_N)`; M3 makes a
genuine `push!`-grown `Vector` compile, which is the corpus's TJ1 and the
"toy → real" milestone for Julia heap support.

**Architecture (3+1 synthesis).** Both architects converged: M3 lives inside
the `has_growend` branch of `_detect_gc_preamble!`, reuses M2's
`_partition_skeleton` / `_build_rerooted_slice` / soundness obligations, infers
capacity from the element-store offset set, and adds a growend!-diamond

M2/M1 byte-identical with the singleton `Set([data_ptr])`) — chosen over a
threaded `base_canon` closure (data, not a function, is auditable under
debugging). (3) Phis classified by whether incomings are skeleton pointers vs
a value — NOT a hard "==2 phis" assertion (the final JOIN carries 2 skeleton
ptr-phis *plus* the reduce value-phi).

**Implementation shape.** `_recognise_growend_array` builds `_HeapArray`
structurally; `_collapse_growend_diamond` recognises + collapses each
`@j_#_growend!` capacity diamond (non-mutating — produces a `dropped` set);
`_infer_growend_capacity` derives N from the dense element-store offset prefix;
`_m3_skeleton` rebuilds the taint closure with element-data loads as SINKS;
`_m3_value_phi_redirects` resolves the reduce value-phi; `_prove_growend_partition_sound`
runs M2's O1-O7 (`data_roots`-aware) plus M3's G1/G3/G4; `_m3_compile`
orchestrates. Routing: the `has_growend` branch now discriminates `heap_live`
→ M3 path, else the unchanged M1 path.

**Gotchas / lessons:**
- **IR recon was materially wrong; the implementer re-extracted and caught it.**
  The design recon said f_tj1 has a Memory `@ijl_gc_small_alloc`. It does not —
  the only `gc_small_alloc` is the 32-byte Array WRAPPER; the empty `Int8[]`
  Memory backing is Julia's shared empty-Memory module global
  (`@"jl_global#NNN.jit"`, capacity 0), and the real Memory materialises only
  inside `growend!`'s realloc. So M3 cannot reuse M2's `_recognise_heap_array`
  (it needs a Memory alloc + constant length store — neither exists);
  `_recognise_growend_array` builds the `_HeapArray` from scratch. **Lesson:
  the implementer re-extracting the IR (CLAUDE.md §5/§9) before writing code
  caught three divergences from the recon — it STOPPED and reported rather than
  coding against a wrong model. That is the senior-engineer move and it is why
  step 0 of any heap milestone must be a fresh `code_llvm`.**
- **D3 — taint over-propagation (the worst-class bug, caught in review of the
  design, not the code).** The standard `_recognise_skeleton` forward closure
  folds an element load's *result-consumers* into SKEL (the load addresses a
  skeleton pointer). For M3 that means the `reduce` adds + the `ret` value get
  tainted → classified `gc_machinery` → DROPPED by `_build_rerooted_slice` → a
  silent miscompile of the entire return. M2 never hit this because M2's
  element load feeds `ret` directly and `ret` is not a taint target — there was
  no load-result *consumer*. Fix: `_m3_skeleton` rebuilds the closure with
  element-data loads as taint SINKS — in SKEL (so the partition captures them),
  but their result does not propagate. BACKSTOP: M2-O3 (machinery-dead) runs on
  the M3 path — if the sink fix were ever wrong and a reduce `add` landed in
  machinery, O3 rejects loud (a machinery instruction that is a ret-ancestor).
  The fix makes it compile; O3 ensures a *broken* fix fails loud, not silent.
- **LLVM load-PRE hoists a reduce element load into a growend! SLOW arm.** The
  index-1 reduce load appears in the L68 SLOW predecessor as
  `%memoryref_data.phi.trans.insert = GEP(reload,1); load i8`. The first cut of
  `_collapse_growend_diamond` dropped *every* SLOW-arm instruction → the hoisted
  live element load was dropped → dangling SSA. Fix: `_m3_is_slow_arm_elem_data`
  — element-data instructions inside a SLOW arm are LIVE traffic, kept for the
  partition; SLOW-arm purity still holds (they are in `skel`).
- **`mem_obj === wrapper` sentinel collision.** M3 sets `_HeapArray.memory_obj`
  to the Array wrapper (a sentinel — M3 never addresses the raw Memory object).
  M2's helpers have a `gi.base === memory_obj && coff≥16 → data region` rule,
  but `GEP(wrapper,16)` is the *size counter*, not data. Added
  `_heap_mem_obj_is_real(harr)` and gated all five `memory_obj`-relative rules;
  M2 always has `memory_obj !== wrapper` so it is a no-op for M2.
- **`LLVM.Value(operand_ref)` CRASHES on GlobalAlias operands** (`store ptr
  @"jl_global", ...`). Read the store-value width via raw `LLVMTypeOf`
  (`_m3_raw_int_width`), never `LLVM.Value(...)`. Same landmine M1/M2 hit; heap
  code uses the raw C-API operand accessors throughout.
