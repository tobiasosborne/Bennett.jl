**Gotchas / lessons:**
- **IR recon was materially wrong; the implementer re-extracted and caught it.**
  The design recon said f_tj1 has a Memory `@ijl_gc_small_alloc`. It does not —
  the only `gc_small_alloc` is the 32-byte Array WRAPPER; the empty `Int8[]`
  Memory backing is Julia's shared empty-Memory module global
  (`@"jl_global#NNN.jit"`, capacity 0), and the real Memory materialises only
  inside `growend!`'s realloc. So M3 cannot reuse M2's `_recognise_heap_array`
  (it needs a Memory alloc + constant length store — neither exists);
  `_recognise_growend_array` builds the `_HeapArray` from scratch. **Lesson:
  the implementer re-extracting the IR (CLAUDE.md §5/§9) before writing code
  caught three divergences from the recon — it STOPPED and reported rather than
  coding against a wrong model. That is the senior-engineer move and it is why
  step 0 of any heap milestone must be a fresh `code_llvm`.**
- **D3 — taint over-propagation (the worst-class bug, caught in review of the
  design, not the code).** The standard `_recognise_skeleton` forward closure
  folds an element load's *result-consumers* into SKEL (the load addresses a
  skeleton pointer). For M3 that means the `reduce` adds + the `ret` value get
  tainted → classified `gc_machinery` → DROPPED by `_build_rerooted_slice` → a
  silent miscompile of the entire return. M2 never hit this because M2's
  element load feeds `ret` directly and `ret` is not a taint target — there was
  no load-result *consumer*. Fix: `_m3_skeleton` rebuilds the closure with
  element-data loads as taint SINKS — in SKEL (so the partition captures them),
  but their result does not propagate. BACKSTOP: M2-O3 (machinery-dead) runs on
  the M3 path — if the sink fix were ever wrong and a reduce `add` landed in
  machinery, O3 rejects loud (a machinery instruction that is a ret-ancestor).
  The fix makes it compile; O3 ensures a *broken* fix fails loud, not silent.
- **LLVM load-PRE hoists a reduce element load into a growend! SLOW arm.** The
  index-1 reduce load appears in the L68 SLOW predecessor as
  `%memoryref_data.phi.trans.insert = GEP(reload,1); load i8`. The first cut of
  `_collapse_growend_diamond` dropped *every* SLOW-arm instruction → the hoisted
  live element load was dropped → dangling SSA. Fix: `_m3_is_slow_arm_elem_data`
  — element-data instructions inside a SLOW arm are LIVE traffic, kept for the
  partition; SLOW-arm purity still holds (they are in `skel`).
- **`mem_obj === wrapper` sentinel collision.** M3 sets `_HeapArray.memory_obj`
  to the Array wrapper (a sentinel — M3 never addresses the raw Memory object).
  M2's helpers have a `gi.base === memory_obj && coff≥16 → data region` rule,
  but `GEP(wrapper,16)` is the *size counter*, not data. Added
  `_heap_mem_obj_is_real(harr)` and gated all five `memory_obj`-relative rules;
  M2 always has `memory_obj !== wrapper` so it is a no-op for M2.
- **`LLVM.Value(operand_ref)` CRASHES on GlobalAlias operands** (`store ptr
  @"jl_global", ...`). Read the store-value width via raw `LLVMTypeOf`
  (`_m3_raw_int_width`), never `LLVM.Value(...)`. Same landmine M1/M2 hit; heap
  code uses the raw C-API operand accessors throughout.

**Review (+1) findings, all resolved:**
- **G2 was dead code — deleted.** The synthesised design specified an
  obligation G2 (pointer-family no-escape). Review found it unreachable:
  `_prove_growend_partition_sound` runs M2's obligations first, and M2-O5 —
  which the `data_roots` refactor generalised to `union!(array_refs,
  harr.data_roots)` — already scans the whole buffer-base family for escape.
  G2 iterated a strict subset; any escape trips M2-O5 first. G2 deleted; escape
  is M2-O5's job (the `data_roots` refactor was *designed* to make O5 the one
  general escape obligation). The proof's obligations are now G1/G3/G4 + the
  carried-over M2 O1-O7; the G1→G3 label gap is documented in-file.
- **Reject tests now pin the failure reason.** The 7 reject testsets were
  bare `@test_throws ErrorException` — strengthened with message-substring
  assertions (`_m3_assert_rejects`) so a future regression that rejects for the
  *wrong* reason is caught.

  attributed to M2-O5 — an M3-shaped (`data_roots`-family) escape, confirmed to
  reject with the M2-O5 message.
